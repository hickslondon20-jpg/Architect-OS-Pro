## Verification tasks (each → a section in 01-01-DELTA.md, ending CONFIRMED / CORRECTED / RISK)

### A. `ose_knowledge_pages` storage
- Full column/type/constraint/index/RLS dump (live). Row count + any existing data.
- The `pinecone_vector_id` situation: is there any pgvector embedding column, or only the legacy Pinecone
  handle? Determine what page-embedding path exists vs. what must be added (pgvector).
- `canonical_key`, `page_kind`, `domain`, `confidence`, `status`, the three date fields — document their
  intended semantics (from code usage).

### B. Taxonomy + source types
- Enumerate `PageType`, `PAGE_TYPE_LABELS`, `WIKI_CATEGORIES`, `STARTER_PAGE_TYPES`, `IMPORT_SOURCES`
  (`osEngineMockData.ts`). Is the taxonomy real/wired or mock placeholder? How does it compare to
  theafh's entity/concept/comparison/summary/query/procedure?
- **Layer 2 is multi-source (CONTEXT §1):** documents + CSO threads + sprint history + domain-agent
  artifacts. Inspect `IMPORT_SOURCES` and the scaffold for what source types it already contemplates —
  does it anticipate non-document sources (threads, sprints, agent artifacts), or documents only? Note
  whether `page_type`/`category` distinguish a sprint-history/longitudinal page category.

### C. UI surfaces
- `WikiView`, `IndexView`, `ManifestView`, `LogView`, `Reader`, `NotesComposer`/`StructureRail`/`FileNode`:
  what each binds to (live `ose_*` tables vs `osEngineMockData`). What is functional vs. empty scaffold.
- The `ManifestView` file→page map and `IndexView` catalog — confirm they're the index/log surfaces the
  emergent wiki needs.

### D. Virtual CSO read-hook
- How `api/vcso/chat.ts` queries/uses `ose_knowledge_pages` (what it selects, when, how injected into the
  prompt). This is the existing partial CSO wiring — document it precisely (it informs the connection-phase boundary).

### E. The ingest gap (the core question)
- Confirm **no** synthesis/ingest engine writes `ose_knowledge_pages` (search `lib`, `python-backend`,
  `api`, edge functions, n8n references). What does `seed_core_knowledge_pages()` actually create?
- Search for any abandoned/partial page-generation code (a clue to why it stalled).

### F. Related OSE tables
- `ose_page_corrections` (override surface), `ose_activity_log` (the log), `ose_knowledge_base_setup`
  (onboarding), `ose_raw_document_registry` (raw uploads). Columns + how they interrelate with
  `ose_knowledge_pages` (esp. `source_file_ids` ↔ registry = the manifest).

### G. The document source
- Confirm the KB Explorer (`kb_folders`, `full_markdown` in `ose_raw_document_registry`, the ls/tree/grep/
  read tools) is the document input the synthesis engine would read from. Note `document_chunks` (Tier 2).

### H. Reconciliation with Layer 1 (`wiki_*`)
- How `ose_knowledge_pages` relates to the 7 fixed `wiki_*` pages. The Layer-1 schema object carries an
  `ose_page_type` mapping per fixed page — document the intended Layer-1↔Layer-2 boundary so they
  complement (breadth vs depth), not duplicate.

### I. Build-on-existing verdict
- Synthesize A–H into an explicit recommendation: **build the engine on `ose_knowledge_pages`** (validate
  the prior) **or** stand up a reconciled/fresh store (challenge it) — with reasons. Flag any legacy
  entanglement (Pinecone, mock-vs-wired UI) that affects the call.

## Deliverable
`01-01-DELTA.md` (this folder), sections A–I with verdicts, a **gap table** (what exists / what the
synthesis engine must add), and the §I recommendation. Any finding that changes a `../../CONTEXT.md` open
decision is noted for the design discuss.

4. Ingest gap confirmed; any abandoned page-generation code surfaced.
5. KB Explorer source + Layer-1 boundary documented.
6. Explicit build-on-existing-vs-fresh recommendation with reasons. No production code written.
