# Phase 0 Plan 02 — Wiki Authority Decision + Conversation-Feeder Verification

> Read the workstream `../../CONTEXT.md` + `../../ROADMAP.md` and this phase's `00-CONTEXT.md` first.
> Covers **CLEAN-3, CLEAN-4**. Evidence base: `../../../RECONCILIATION-COGNITIVE-ORCHESTRATION.md`.
> **Investigate and recommend — the authority choice is a founder decision (do not unilaterally pick).**
> No page authoring, no OS Engine changes, no router build.

## Deliverable
A findings note that (1) maps the two wiki representations and recommends the authoritative query-time
source + read path for founder confirmation, and (2) states whether the conversation→wiki feeder runs
or is scoped deferred. Written into `00-COMPLETION.md` and the workstream STATE; no code that changes
behavior.

## Steps

### A. Map the two wiki representations (CLEAN-3)
1. **`ose_knowledge_pages`** — the pages the live VCSO prompt actually loads (`_build_context` →
   `_load_founder_context` selects a keyword-scored slice, core-key boosted). Document its schema
   (page_type/canonical_key/content), the 7 Layer-1 core keys, and how Layer-2 pages appear here.
   Confirm counts per test founder (reconciliation: 12 for `cd490873…`).
2. **`wiki_*` family** — `wiki_pages`, `wiki_claims`, `wiki_evidence`, `wiki_digest`,
   `wiki_insight_records`, `wiki_contradictions`, plus `wiki_schema`. Document what each holds and how
   the `per_user_wiki` sub-agent reads them (`_handle_per_user_wiki` — `wiki_get_page`/`wiki_get_claim`/
   `wiki_search`/`wiki_search_insight`/`wiki_read_digest`). Confirm population per founder.
3. Map the relationship: are these **two independent systems**, one **derived from** the other, or
   **divergent duplicates**? Identify which OS Engine writer populates each (`wiki_writeback.py`,
   `wiki_compilation.py`, `doc_wiki_*`).
4. **Recommend** the authoritative query-time source for composition (North Star "pre-reasoned
   components") + how the future composer should read it — with a one-paragraph rationale. Flag
   explicitly **for founder confirmation** (this resolves Conflict O2 and shapes P1/P3).

### B. Verify the conversation→wiki feeder (CLEAN-4)
1. Trace whether a VCSO thread produces a synthesis record that feeds OS Engine to create/evolve a
   Layer-2 page (per architecture §5). Check for the path: VCSO thread → synthesis record → workflow
   (N8N or Python) → OS Engine writeback. Inspect `wiki_writeback.py`, `wiki_compilation.py`,
   `doc_wiki_cso_thread_adapter.py`, and any scheduled/N8N trigger.
2. Evidence test: do any `ose_knowledge_pages` / `wiki_*` rows trace to a **chat thread** origin
   (vs. document upload)? Are there recent thread-sourced pages, or do all pages track uploads?
3. Conclude: **running** (with evidence) **or deferred dependency** (with a one-paragraph rationale —
   it does not block Phase 1, but Phase 3's build-time cost lever depends on it, so its status is
   recorded as an explicit assumption).

## Acceptance criteria
1. A documented map of `ose_knowledge_pages` vs. `wiki_*` (contents, which the VCSO reads, who writes
   each, and their relationship), with per-founder population confirmed.
2. A **recommended authoritative query-time wiki + read path**, rationale included, flagged for founder
   confirmation (O2).
3. Feeder status **confirmed running (evidence)** or **scoped deferred (rationale)** (O3).
4. Findings written to `00-COMPLETION.md`; workstream `STATE.md` + `../../../RECONCILIATION-COGNITIVE-ORCHESTRATION.md`
   notes updated; the workstream `CONTEXT.md` Conflict Register O2/O3 marked resolved/scoped once the
   founder confirms.
5. No behavior-changing code committed in this plan (investigation + documentation only).

## Out of scope
Authoring the new founder-operating pages (OS Engine dependency); building the router/assembly (P1/P3);
building or modifying the feeder or any OS Engine writer; changing which wiki the live VCSO reads (that
is a later, deliberate phase change once authority is confirmed).
