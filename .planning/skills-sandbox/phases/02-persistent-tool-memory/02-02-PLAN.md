## Goal

Wire the places that currently discard or never fetch persisted tool-call data, so:
(a) a reopened thread shows its historical `agentSteps` instead of losing them silently,
(b) a follow-up question in the same thread can draw on a prior tool result without re-invoking it,
and (c) — **confirmed in scope by London at the checkpoint** — the nested sub-agent "thinking"
section actually streams live above the response *during generation*, which was never wired at all,
independent of the reload problem. (a)/(b) and (c) share one root cause: `agentSteps` are computed
early but only ever transmitted in the final `done` SSE event, once the whole response has already
streamed.

This plan touches three files. No new tables, no new endpoints.

## Changes

### 1. `api/vcso/chat.ts` — `callKbExplorer()` return shape

Currently returns `{ resultSummary, steps }`, discarding `data.run_id` from the Python backend's
`AgentRunResponse`. Add `runId`:

```ts
return { resultSummary, steps, runId: data.run_id ?? null };
```
Update the local type: `{ resultSummary: string; steps: AgentStep[]; runId: string | null } | null`.

### 2. `api/vcso/chat.ts` — backfill `assistant_message_id` after the assistant message insert

Immediately after the existing `assistantMessage` insert succeeds (~line 604), add:

```ts
if (kbResult?.runId) {
  await supabase
    .from('agent_delegation_runs')
    .update({ assistant_message_id: assistantMessage.data.id })
    .eq('id', kbResult.runId)
    .eq('user_id', userId);
}
```
Use the founder's own `supabase` client (not `service`) — RLS already scopes this correctly to the
run's owner, consistent with every other write in this handler.

### 3. `api/vcso/chat.ts` — fold persisted tool results into follow-up context

Extend the `recentResult` query (~line 512) — currently `.select('role,content,created_at')` — so
that for any assistant message in the window, matching `agent_delegation_steps` (joined via
`agent_delegation_runs.assistant_message_id`) are also fetched and passed into `assemblePrompt()`.
Concretely: after `recentResult` resolves, collect the assistant message ids in that window, run one
additional query —

```ts
const { data: pastRuns } = await supabase
  .from('agent_delegation_runs')
  .select('assistant_message_id, agent_delegation_steps(tool_name, status, input_summary, output_summary)')
  .in('assistant_message_id', assistantMessageIdsInWindow)
  .eq('user_id', userId);
```
(exact Supabase nested-select syntax to confirm during execution — the point is one extra query, not
N+1 per message) — and pass the reconstructed `{tool, input, output, status}` list per message into
`assemblePrompt()` as new context (e.g. `priorToolResults`), so the model can answer "what was that
number again" from persisted structured data rather than only the prior assistant message's prose.
**Do not change `assemblePrompt()`'s existing `recentMessages` shape** — add this as a new, separate
input so nothing else consuming `recentMessages` breaks.

### 4. `lib/virtualCsoApi.ts` — `getMessagesForChat()` + `toMessage()`

`getMessagesForChat()` (~line 164) currently does `select('*')` on `vcso_chat_messages` only. Extend
it to also fetch, for assistant-role rows in the result, matching `agent_delegation_steps` via
`agent_delegation_runs.assistant_message_id` (same join shape as change #3, reused — do not write a
second, differently-shaped query). Pass the reconstructed steps into `toMessage(row, steps)`, and
update `toMessage()` (~line 95) to accept and set `agentSteps` from the passed-in value instead of
the dead `row.agentSteps ?? undefined` read (which never resolves to anything real — `02-RESEARCH.md`
§3).

### 5. KB Explorer `agentSteps` reload (resolves ROADMAP Phase 2 success criterion #4)

No separate code path needed — once #3/#4 are wired generically (by `assistant_message_id`, not by
which capability produced the run), KB Explorer's own delegation runs flow through the same
reconstruction automatically. Verify this explicitly as part of testing (see Success Criteria #3
below) rather than assuming it "just works."

### 6. Live-streaming fix (resolves ROADMAP Phase 2 success criterion #5 — confirmed by London)

**Root cause:** `kbResult` (holding `steps`) is fully resolved at ~line 501 — before the `ready` SSE
event fires (~line 560) and well before `streamAnthropic()` starts streaming assistant text
(~line 588). But `agentSteps` is currently attached only to the final `done` event's
`assistantMessage` (~line 646), sent after the entire text stream has already completed. The nested
"thinking" section (`AgentStepsPanel`, built in KB Explorer Phase 9) therefore cannot appear before
or alongside the streaming response today — the data it needs has been sitting unused in scope the
whole time.

**Fix — emit `agentSteps` early:**
- Add `agentSteps: kbResult?.steps ?? undefined` to the existing `ready` SSE payload (~line 560),
  alongside `route`/`assembledContext`. (A separate dedicated event is acceptable if you find a
  concrete reason during implementation to prefer it — e.g. payload-size or client-handling reasons —
  but adding to `ready` is the default: no new event type for the client to learn.)
- Find the client-side streaming-state assembly this needs to feed (per `CONTEXT.md` §5: likely
  `VirtualCSOWorkspace.tsx`'s `onReady`/`onToken`/`done` handling, per
  `.planning/knowledge-base-explorer/phases/09-retrieval-router/09-RESEARCH.md` §10 — confirm the
  actual location, don't assume it hasn't moved). Render `AgentStepsPanel` as soon as the early
  `agentSteps` arrive via `onReady`, **above** the streaming text bubble that fills in shortly after
  via `onToken` — this is the actual "streamed in a nested section above the returned message"
  behavior London described.
- **Keep the `done` event's `agentSteps` as-is** — it remains the source of truth for the final,
  committed `Message` object (what gets pushed into chat state and is what reload later reproduces).
  The early `ready`-event emission is a UI-timing improvement only; do not remove or alter the `done`
  payload's existing `agentSteps` field.
- One `AgentStep[]` shape, sent twice (early for live display, final for persisted state) — do not
  invent a second shape for the early emission.

## Success Criteria

1. Sending a message that triggers `callKbExplorer()` results in `agent_delegation_runs.
   assistant_message_id` being populated for that run (verify with a direct query after a live test
   call, once `ARCHITECTOS_PYTHON_BACKEND_URL`/`ARCHITECTOS_INGEST_SECRET` are confirmed set — if
   they're unset in this environment, note it and verify the code path compiles/type-checks
   correctly instead of skipping verification entirely).
2. Reloading a thread (`getMessagesForChat()`) that contains a message with a linked
   `agent_delegation_run` reconstructs that message's `agentSteps` correctly — matches what was shown
   live during the original turn, not empty.
3. A KB-Explorer-triggered message's `agentSteps` survive a reload with no KB-Explorer-specific code
   change required — confirms #5 above.
4. A follow-up message in the same thread, sent after a tool-call turn, receives the prior tool
   result as part of its assembled prompt context (inspect the assembled prompt in a test call —
   don't just check that the follow-up "seems to work," confirm the structured data is actually
   present in the prompt sent to Claude).
5. No regression to `assemblePrompt()`'s existing `recentMessages`-based behavior for threads with no
   tool calls at all (the common case — most messages have no associated `agent_delegation_run`).
6. **(Confirmed by London)** During a live KB-Explorer-triggering test call, `AgentStepsPanel` renders
   above the message content *before or during* text streaming — not only after the message finishes.
   Confirm by observing (or logging) the client receiving `agentSteps` on the `ready` event, ahead of
   the first `onToken` chunk, not only on `done`.
