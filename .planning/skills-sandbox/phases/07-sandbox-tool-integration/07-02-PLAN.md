## Part A — Trigger: `callSandboxExecution`

In `api/vcso/chat.ts`, after `loadSelectedSkillBodies` resolves (line ~650, `selectedBodies.packs`
— confirmed `select('*')` at line 249, so `requires_sandbox` is already present with no query
change):

```ts
const needsSandbox = selectedBodies.packs.some((pack: any) => pack.requires_sandbox === true);
const sandboxResult = needsSandbox
  ? await callSandboxExecution(userId, text, threadId, userMessage.id)
  : null;
```

New `callSandboxExecution`, mirroring `callKbExplorer` (lines ~300-340) exactly in structure:
same env vars (`ARCHITECTOS_PYTHON_BACKEND_URL`, `ARCHITECTOS_INGEST_SECRET`), same 10-second
`AbortController`, same `catch { return null }` / `finally { clearTimeout(timeout) }` shape. The
POST body differs from `callKbExplorer`'s only in `capability_key: "sandbox_execution_agent"` and
**must include `context_scope: { thread_id: threadId }`** — per `CONTEXT.md` §4's confirmed finding
that this is the only path `_handle_sandbox_execution` has to the sandbox session's thread id.

Thread `sandboxResult` into `assemblePrompt` (mirrors `kbFindings`, line ~433/488): add a
`sandboxFindings?: string` param and a corresponding prompt section (e.g. `` `SANDBOX EXECUTION
RESULT\n${args.sandboxFindings ?? 'Not invoked for this message.'}` ``).

## Part B — Live-turn artifact delivery

07-01's `_handle_sandbox_execution` result dict includes an `artifact_id` when a file was
delivered (see `CONTEXT.md` §4 / `07-01-PLAN.md` Part C). After the assistant message is persisted
and `agent_delegation_runs.assistant_message_id` is set (mirroring the existing KB Explorer
write-back at chat.ts lines ~727-731), also write the artifact id into that same run's
`structured_result` (e.g. `{ artifact_id: sandboxResult.artifactId }`) — **do not create a new
column or table for this linkage; `agent_delegation_runs.structured_result` already exists and is
already read by `loadPriorToolResults`** (line 340-344 selects `structured_result` today but
nothing currently reads it — confirm this before assuming, the field may be unused rather than
absent).

For the *current* SSE turn (founder is watching it stream live, not reloading the thread), the
assistant's final message payload sent over SSE needs an `artifactId` (or full `ArtifactDelivery`
object — cheaper to send the id and let the client call `getArtifact`, matching how sources/pages
are already handled) so the frontend doesn't have to wait for a reload to see the card. Add this to
the `chat`/final SSE event payload alongside the existing `assistantMessage`/`sources`/
`sourcePages` fields.

In `lib/virtualCsoApi.ts`, thread the artifact id through: extend `toMessage()` to accept an
optional `artifacts?: ArtifactDelivery[]` third param (mirroring the existing `agentSteps` second
param) and set `artifactDeliveries: artifacts`. In the live-turn SSE handler (around line 325,
`assistantMessage = toMessage(payload.assistantMessage)`), if the payload carries an artifact id,
call `getArtifact(artifactId)` (already built, `lib/artifactsApi.ts`) and pass the result in.

## Part C — Thread-reload artifact delivery

In the thread-history loader (`lib/virtualCsoApi.ts` lines ~195-229), mirror the existing
`stepsByMessageId` pattern exactly for artifacts: the same `agent_delegation_runs` query at line
210-214 already runs per thread reload — add `structured_result` to its `.select(...)` (it selects

artifactsByMessageId.get(row.id))` at line 229.

## Success Criteria

1. A test skill flagged `requires_sandbox = true`, invoked in a real chat turn, triggers
   `callSandboxExecution` — confirmed via a live thread, not a unit test in isolation.
2. The sandbox result (text) is reflected in the outer Claude call's streamed response for that
   turn.
3. A task that produces a file results in a real `ArtifactDeliveryCard` appearing on the correct
   assistant message *during the live turn* (no reload required).
4. Reloading the same thread (full page refresh or re-navigating to it) shows the same
   `ArtifactDeliveryCard` on the same message — proving the persisted linkage
   (`agent_delegation_runs.structured_result.artifact_id` → `assistant_message_id`) round-trips
   correctly, not just the live-turn SSE path.
5. A follow-up founder question in the same thread, asking about the prior result, is answered
   using `loadPriorToolResults` without re-invoking the sandbox — Success Criterion 5 from the
   Roadmap, verified by checking the network/logs show no second `sandbox_execution_agent` run for
   that follow-up.
6. A skill *not* flagged `requires_sandbox` never triggers `callSandboxExecution` — confirmed
   negative case, not just the positive path.
