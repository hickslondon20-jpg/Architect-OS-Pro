```sql
  ADD COLUMN IF NOT EXISTS requires_sandbox BOOLEAN NOT NULL DEFAULT false;

INSERT INTO public.agent_capabilities (
  capability_key, label, description, status,
  allowed_surfaces, allowed_tools, allowed_source_kinds,
  model_setting_key, output_schema, default_config, can_spawn_agents
) VALUES (
  'sandbox_execution_agent',
  'Sandbox code execution',
  'Runs founder/platform-data code and document generation inside a bounded sandbox session, with access to attached skill files.',
  'experimental',
  ARRAY['virtual_cso'],
  ARRAY['execute_code', 'read_skill_file'],
  ARRAY[]::text[],
  'sandbox_execution_agent',
  '{"version": "agent_result_v1"}'::jsonb,
  '{"max_rounds": 6, "timeout_seconds": 90}'::jsonb,

```

`max_rounds: 6` / `timeout_seconds: 90` are proposed defaults, slightly higher than
`kb_explorer_agent`'s `5`/`60` since code execution (write → see error → fix → rerun) plus GKE pod
interaction is plausibly slower per round than a KB read call. Treat as a starting point — tune
based on real timing observed in Part C's verification pass, not as a fixed requirement.

## Part B — `SandboxExecutionService`

New `python-backend/services/sandbox_execution_service.py`, mirroring
`kb_explorer_service.py`'s shape exactly:

```python
SANDBOX_EXECUTION_SYSTEM_PROMPT = """You are a sandboxed code-execution agent for ArchitectOS Pro.
Your job is to write and run Python code against founder/platform data to answer a calculation or
document-generation task, using any attached skill files as reference or template material.

Available tools:
- execute_code: Run Python code in a persistent sandbox session. State (variables, imports)
  persists across calls within this task.
- read_skill_file: Read the full content of a file attached to the skill guiding this task.

Rules:
- If code raises an error, read the error and fix it — do not give up after one failed attempt.
- Prefer producing a real output file (via python-docx, python-pptx, openpyxl, pandas, matplotlib,
  etc.) when the task calls for a document or spreadsheet, not just printed text.
- Be concise in your final summary. State what was produced and where relevant.
"""

SANDBOX_EXECUTION_TOOLS = [
    {
        "name": "execute_code",
        "description": (
            "Run Python code in the persistent sandbox session for this task. Variables and "
            "imports persist across calls. Returns stdout, stderr, and exit code. If your code "
            "writes an output file, report its path in your final summary so it can be delivered."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "code": {"type": "string", "description": "Python code to execute."},
            },
            "required": ["code"],
        },
    },
    {
        "name": "read_skill_file",
        "description": (
            "Read the full text or binary-decoded content of a file attached to the skill guiding "
            "this task. Use the file id from the task context."
        ),
        "input_schema": {
            "type": "object",
            "properties": {
                "skill_file_id": {"type": "string", "description": "UUID of the skill file to read."},
            },
            "required": ["skill_file_id"],
        },
    },
]


@dataclass(frozen=True)
class SandboxExecutionResult:
    summary: str
    tool_steps: list[dict[str, Any]]
    produced_file_path: str | None   # container-side path, if any, for ArtifactService handoff
    rounds_used: int
    truncated: bool = False


class SandboxExecutionService:
    def __init__(self, sandbox_service: SandboxService) -> None: ...

    @classmethod
    def from_env(cls) -> "SandboxExecutionService": ...

    def run_execution(
        self,
        user_id: str,
        thread_id: str,
        task_summary: str,
        max_rounds: int = 6,
    ) -> SandboxExecutionResult:
        """Mirrors KbExplorerService.run_exploration's round-capped tool-use loop exactly.
        execute_code dispatches to self._sandbox_service.get_active_session(thread_id) (create the
        session if none is active for this thread yet — first call in a thread pays GKE cold-start
        cost, matching Phase 5/6's established session semantics) then .execute_command(code).
        read_skill_file dispatches to a Supabase query against skill_files by id + Storage download.
        Track any file path the model reports producing (from its final summary or a structured
        convention TBD by the execution agent — e.g. asking the model to state produced files in a
        consistent tagged format) so the caller can hand it to ArtifactService.deliver_from_sandbox.
        """
```

**Note on `produced_file_path` detection:** unlike KB Explorer (which only ever returns
information, never produces files), this loop may end with a file sitting in the sandbox pod that
needs delivery. The execution agent must decide a concrete, reliable convention for the model to
report this (e.g., instruct it in the system prompt to end with a line like
`PRODUCED_FILE: /sandbox/output.xlsx` when it creates a deliverable file, and parse that
deterministically) — do not rely on freeform text parsing of the summary.

## Part C — Orchestrator wiring

`python-backend/services/sub_agent_orchestrator.py`:

- Add `elif capability.capability_key == "sandbox_execution_agent": result =
  self._handle_sandbox_execution(context)` to `start_run` (mirrors the `kb_explorer_agent` branch
  at lines 89-90).
- New `_handle_sandbox_execution(self, context: AgentContextBundle) -> dict[str, Any]`, mirroring
  `_handle_kb_explorer` (lines 318+): read `thread_id = context.context_scope.get("thread_id")`
  (raise a clear `SubAgentError` if absent — this is a caller contract, not optional), call
  `SandboxExecutionService.from_env().run_execution(user_id=context.user_id, thread_id=thread_id,
  task_summary=context.task_summary, max_rounds=<from capability.default_config>)`, and — if
  `produced_file_path` is set — call `ArtifactService.from_env().deliver_from_sandbox(user_id,
  thread_id, produced_file_path, ...)` before building the handler's return dict, so a produced
  file is already delivered (Storage upload + `artifacts` row) by the time the handler returns.
  Fold the resulting artifact id into the handler's `findings`/result dict so the frontend can
  reference it (07-02 needs a way to know an artifact was produced, not just a text summary).
- Add `sandbox_execution_agent` to `agent_capabilities.py`'s `_fallback_capabilities()` matching
  Part A's migration row exactly (keep both in sync — the fallback is a backup path, not the
  primary one, per how `AgentCapabilityRegistry` reads live rows first).

No new FastAPI route is needed — `POST /api/agent-runs` (main.py line 898) already dispatches
generically by `capability_key`.

## Success Criteria

1. `012_sandbox_execution_agent.sql` applied; `agent_capabilities` has a live
   `sandbox_execution_agent` row; `skill_packs.requires_sandbox` column exists, defaults false.
2. A direct call to `SubAgentOrchestrator.start_run()` with `capability_key:
   "sandbox_execution_agent"` and a `context_scope` containing a real `thread_id` successfully runs
   Python code in a live Phase 5 sandbox session and returns a result — proven against the real
   cluster, not mocked.
3. `read_skill_file` successfully reads a real skill file's content from Phase 1's `skill_files`
   storage within the tool loop (test with a skill that has an attached file).
4. A test task that asks for a generated `.xlsx` or `.md` file completes with `produced_file_path`
   set, and `ArtifactService.deliver_from_sandbox` successfully delivers it (Storage upload +
   `artifacts` row present) — proven end to end, not just that the sandbox code ran.
5. The run persists through Phase 2's `agent_delegation_runs`/`agent_delegation_steps` exactly like
   existing capabilities — verified by querying those tables after a test run, not assumed from the
   dispatch code alone.
6. Sandbox session reuse confirmed: two sequential `execute_code` calls within the same
   `run_execution` invocation (i.e. the model fixing an error and rerunning) share state — a
   variable set in round 1 is readable in round 2 — matching Phase 5's proven session-persistence
   behavior.
