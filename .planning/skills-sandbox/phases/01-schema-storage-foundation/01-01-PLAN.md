## Goal

Rename `ip_skill_packs` to `skill_packs`, add founder ownership and an ownership-derived `scope`,
establish the admin-designation mechanism (`profiles.is_admin`), and enforce structurally that only
the admin's rows can ever carry `scope = 'global'` — all while preserving the 6 existing rows as
global admin content.

-- Phase 1 (Agent Skills & Document Generation Engine): rename ip_skill_packs -> skill_packs,
-- add founder ownership + ownership-derived scope, admin designation on profiles.

-- 1. Admin designation (reuse profiles, per CONTEXT.md #2 — no new single-purpose table)
ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS is_admin BOOLEAN NOT NULL DEFAULT false;

-- Set the admin flag for London's account. Replace <ADMIN_USER_ID> with the real value confirmed
-- in Pre-Execution Check #2 — do not guess.
UPDATE public.profiles SET is_admin = true WHERE user_id = '<ADMIN_USER_ID>';

-- 2. Rename + extend
ALTER TABLE public.ip_skill_packs RENAME TO skill_packs;

ALTER TABLE public.skill_packs
  ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  ADD COLUMN scope TEXT NOT NULL DEFAULT 'private' CHECK (scope IN ('global', 'private'));

-- 3. Backfill the 6 pre-existing rows as global admin content
UPDATE public.skill_packs
SET user_id = '<ADMIN_USER_ID>', scope = 'global'
WHERE user_id IS NULL;

-- 4. Now that every row has an owner, enforce NOT NULL
ALTER TABLE public.skill_packs ALTER COLUMN user_id SET NOT NULL;

-- 5. Structural write-lock: only the admin's rows may carry scope = 'global'
CREATE OR REPLACE FUNCTION public.enforce_skill_pack_global_scope()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.scope = 'global' AND NOT EXISTS (
    SELECT 1 FROM public.profiles WHERE user_id = NEW.user_id AND is_admin = true
  ) THEN
    RAISE EXCEPTION 'skill_packs.scope can only be global for the admin-owned account (user_id=%)', NEW.user_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;

CREATE TRIGGER skill_packs_enforce_global_scope
  BEFORE INSERT OR UPDATE ON public.skill_packs
  FOR EACH ROW
  EXECUTE FUNCTION public.enforce_skill_pack_global_scope();

-- 6. Indexes
CREATE INDEX IF NOT EXISTS skill_packs_user_id_idx ON public.skill_packs (user_id);
CREATE INDEX IF NOT EXISTS skill_packs_scope_idx ON public.skill_packs (scope);
CREATE INDEX IF NOT EXISTS skill_packs_status_idx ON public.skill_packs (status);

-- 7. RLS — replace service-role-only with founder-facing read + owner-scoped write
-- (the existing "Service role can manage IP skill packs" ALL/true policy stays as-is for backend ops)
CREATE POLICY "skill_packs_select_own_or_global"
  ON public.skill_packs FOR SELECT
  USING (scope = 'global' OR user_id = (SELECT auth.uid()));

CREATE POLICY "skill_packs_insert_own"
  ON public.skill_packs FOR INSERT
  WITH CHECK (user_id = (SELECT auth.uid()));

CREATE POLICY "skill_packs_update_own"
  ON public.skill_packs FOR UPDATE
  USING (user_id = (SELECT auth.uid()))
  WITH CHECK (user_id = (SELECT auth.uid()));

CREATE POLICY "skill_packs_delete_own"
  ON public.skill_packs FOR DELETE
  USING (user_id = (SELECT auth.uid()));

## Success Criteria

1. `ip_skill_packs` no longer exists; `skill_packs` exists with all 6 original rows preserved,
   `user_id` and `scope` populated for every row (`scope = 'global'`, `user_id` = admin).
2. `profiles.is_admin` exists; exactly one row (the confirmed admin account) has `is_admin = true`.
3. Attempting `UPDATE skill_packs SET scope = 'global' WHERE id = <a private test row>` (owned by a
   non-admin `user_id`) is rejected by the trigger.
4. A founder JWT can SELECT global rows + their own rows, and cannot SELECT another founder's
   private rows.
5. A founder JWT can INSERT a row for themselves (`scope` defaults to `'private'`) but cannot INSERT
   or UPDATE a row into `scope = 'global'`.
6. `generate_typescript_types` re-run; `skill_packs` (not `ip_skill_packs`) reflected in generated
   types.
7. `api/vcso/chat.ts` is **not modified** — confirm its existing `ip_skill_packs`-referencing queries
   now correctly resolve against `skill_packs` (Postgres rename preserves all references automatically
   at the DB level; if the frontend/backend code references the table name as a literal string
   anywhere outside `api/vcso/chat.ts`'s Supabase client calls — e.g. in raw SQL — those references
   need updating even though this isn't a Phase-1-scoped code change; flag any found instead of
   silently fixing them, since touching `api/vcso/chat.ts` is explicitly Phase 3's job per
   `CONTEXT.md` §4).
