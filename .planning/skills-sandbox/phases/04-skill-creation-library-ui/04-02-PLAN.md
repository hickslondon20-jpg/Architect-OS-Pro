# Plan 04-02 — AI-Guided Skill Creation Flow

**Phase:** 04 — Skill Creation & Skills Library UI
**Plan:** 2 of 3
**Depends on:** 04-01-PLAN.md (`POST /api/skills` — this plan's terminal write path)
**Status:** Ready for execution
**Decisions:** `CONTEXT.md` §2 (this folder) — dedicated surface, not a Virtual CSO chat retrofit

---

## Goal

A structured, in-thread conversational flow — living inside the Skills & Plugins workspace, not
Virtual CSO chat — that walks a founder through defining a skill (name, description, trigger
conditions, instructions) and ends by calling `POST /api/skills` from 04-01. Mirrors the feel of the
Cowork/Claude Code skill-creator experience without requiring a general agentic tool-loop.

---

## Approach

This does not need Virtual CSO's streaming/routing/persistent-memory machinery — it's a bounded,
single-purpose flow with a known end state (one `skill_packs` row). Recommend: a lightweight,
purpose-built chat-like UI component (reuse message-bubble styling patterns from
`components/pro-suite/virtual-cso/MessageBubble.tsx` for visual consistency, but not the underlying
streaming/SSE/thread-persistence infrastructure) backed by a simple multi-turn Claude call (via
whatever the existing non-streaming Claude call pattern is elsewhere in the codebase — check
N8N-routed synthesis calls per `CLAUDE.md`'s "all synthesis routes through N8N except the Virtual CSO
streaming exception" rule; **this flow is not the Virtual CSO streaming exception, so it should route
through N8N like other synthesis, not get its own new direct Anthropic call path** — confirm this
against `CLAUDE.md`'s architecture rule before implementing, since it's an explicit, named
architectural constraint, not a default to assume away).

The conversation accumulates a draft skill definition turn by turn; the founder confirms before the
final `POST /api/skills` call fires — no silent auto-save partway through a conversation.

---

## Success Criteria

1. A founder can complete this flow end to end and produce a valid `skill_packs` row via `POST
   /api/skills` (04-01) — not a separate, parallel write path.
2. The flow does not touch `api/vcso/chat.ts` or Virtual CSO's streaming/thread infrastructure at all.
3. Synthesis calls in this flow route through N8N, per `CLAUDE.md`'s architecture rule — confirmed,
   not assumed. If a genuine reason emerges to need direct streaming here too, that's a new
   architectural exception to flag explicitly (mirroring how the Virtual CSO streaming exception
   itself was a deliberate, documented decision) — not something to add quietly.
4. The founder explicitly confirms before the skill is created — no partial/silent saves.
5. Visual style matches ArchitectOS design tokens (per `DESIGN-GUIDE-QUICK.md`), reusing
   `MessageBubble.tsx`'s visual conventions where sensible without importing Virtual CSO's
   streaming/persistence code.

## Out of Scope

`POST /api/skills`'s implementation (04-01, already built). The Skills & Plugins workspace shell this
flow lives inside (04-03 builds the shell/routing; this plan is the flow itself, which 04-03 mounts).
