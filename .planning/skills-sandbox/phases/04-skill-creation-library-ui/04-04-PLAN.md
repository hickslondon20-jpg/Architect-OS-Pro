**Status:** Confirmed by London 2026-07-01, ready for execution agent.
**Depends on:** 04-01, 04-02, 04-03 (all complete and independently verified against this plan's two gaps).
**Why this exists:** After the Phase 4 execution agent finished, London corrected two assumptions baked into

## Part A — Migrate the AI-guided skill draft off the N8N webhook dependency

### Current state (verified 2026-07-01)
`lib/skillsApi.ts`'s `requestGuidedSkillDraft()` (lines 130-148) has a hard split:
- If `VITE_SKILL_CREATOR_WEBHOOK_URL` is **unset**, it returns a static canned message
  (`"Tell me what repeatable judgment, report, or workflow this skill should help your Virtual CSO
  perform."`) and computes `ready` as a plain boolean check on whether `currentDraft.name`,
  `.description`, and `.body` are already present — **there is no LLM involvement at all in this path.**
- If the webhook URL **is** set, it POSTs `{messages, currentDraft}` to an external N8N webhook and
  trusts whatever JSON comes back.

This means the "AI-guided creation flow" — one of Phase 4's three creation paths — has no working AI
conversation unless a separate N8N workflow has been built and its URL configured as an env var. That's a
real functional gap for a beta feature, not just an architectural style question: nothing in this repo
builds that N8N workflow, so today this path most likely serves the degraded fallback in production.

### What to build
Move this synthesis colocated into the Python backend, mirroring the direct-Anthropic pattern already
proven in `python-backend/services/doc_wiki_synthesis.py` and `python-backend/services/kb_explorer_service.py`
(both confirmed live during Phase 4 research) — this is now the sanctioned default per `CLAUDE.md` Rule #1.

1. **New endpoint:** `POST /api/skills/guided-draft` on the `skills` router (`python-backend/`, same router
   mounted at `/api/skills` per `main.py` line 487). Request body: `{ messages: Array<{role, content}>,
   currentDraft: Partial<SkillPayload> }` (same shape the frontend already sends).
2. **Server-side synthesis:** a small service function (new file, e.g.
   `python-backend/services/skill_draft_synthesis.py`, structured like `doc_wiki_synthesis.py`'s
   `_call_synthesis_claude`) that:
   - Builds a system prompt instructing Claude to act as a skill-creation interviewer for ArchitectOS Pro
     founders, extracting: `name`, `description`, `domain`, `skill_kind`, `trigger_tags`,
     `required_platform_context`, `body` (SKILL.md-format markdown per the locked decision in
     `.planning/skills-sandbox/CONTEXT.md`).
   - Passes the conversation history (`messages`) plus `currentDraft` as context.
   - Instructs the model to respond with **only** a JSON object matching `GuidedSkillDraft` (the existing
     frontend type in `lib/skillsApi.ts` lines 33-36, extended with whichever `SkillPayload` fields are now
     confirmed).
   - Parses the model's JSON response server-side; on parse failure, return a safe fallback
     (`assistant_message` asking the founder to rephrase, `ready: false`) rather than a 500 — do not let a
     malformed model response break the conversation.
   - Uses `anthropic.Anthropic(api_key=settings.anthropic_api_key)`, same client construction pattern as
     the two reference services.
3. **Frontend change:** `lib/skillsApi.ts`'s `requestGuidedSkillDraft()` calls
   `${getBaseUrl()}/api/skills/guided-draft` (the same Python backend + auth pattern as every other
   `/api/skills*` call in this file — `getAuthHeaders()`, founder JWT), not the N8N webhook.
4. **Remove the dependency:** delete `SKILL_CREATOR_WEBHOOK_URL` and its canned-fallback branch entirely —
   there is no more degraded path once the direct call is self-contained inside the app's own backend.
   If `VITE_SKILL_CREATOR_WEBHOOK_URL` is referenced anywhere else (env files, deployment config), flag it
   for removal there too, but don't block this plan on cleaning up deployment config you can't see.

### Success criteria
1. A founder can complete a full guided-draft conversation with zero external N8N configuration required.
2. `requestGuidedSkillDraft()` no longer references any N8N webhook URL.
3. The new endpoint's synthesis calls Anthropic directly from the Python backend, per `CLAUDE.md` Rule #1's
   first lane.
4. A malformed/non-JSON model response degrades gracefully (does not 500, does not crash the conversation
   UI) — verify this with at least one deliberately-forced bad-response test, not just the happy path.
5. The resulting draft, once `ready: true`, saves through the existing `POST /api/skills` path unchanged
   (04-01 — do not build a second creation path).

## Part B — ChatRail skill list: search + insert-into-textbox, not just navigation

### Current state (verified 2026-07-01)
`components/pro-suite/virtual-cso/ChatRail.tsx`:
- Lines 107-117: the search input only renders when `railMode === 'chats'` — **no search exists for the
  skills view.**
- Lines 131-143: each skill row is `<Link to="/pro/intelligence/skills">` — clicking it **navigates away**
  to the full workspace. There is no "use skill" affordance and no mechanism to insert `@slug` into a
  compose textbox.

This matches exactly what `04-03-PLAN.md` specified ("browse-only... clicking navigates to full workspace")
— the execution agent built precisely what was asked. The gap is between that plan and London's actual
intent, clarified afterward: a searchable micro-library where selecting a skill inserts its `@slug` into
whatever compose textbox is currently active, so a founder can invoke it in the message they're about to
send without leaving the conversation.

### What to build
1. **Search input for the skills view.** Add a search box above the skill list (same visual treatment as
   the existing chats-mode search, lines 107-117) that filters the already-loaded `skills` array client-side
   by substring match against `name`, `slug`, and `description`. No new backend query needed — `loadSkills()`
   already returns the founder's full scoped list.
2. **"Use skill" action that inserts `@slug` into the compose textbox**, not just a Link-to-navigate:
   - `ChatRail` needs a new callback prop, e.g. `onUseSkill: (slug: string) => void`, passed down from
     whichever parent owns the active compose textbox's state — locate this in
     `pages/ProSuite/virtual-cso/VirtualCSOWorkspace.tsx` (same file that already wires `agentSteps` per
     Phase 2's fix) rather than assuming its shape; read the component's message-input state before wiring
     this.
   - Each skill row's primary click action calls `onUseSkill(skill.slug)`, which inserts `@${slug} ` into
     the active compose textbox and focuses it (append if there's existing text, don't clobber a
     founder's in-progress draft).
   - Keep a way to still reach the full Skills & Plugins workspace for viewing/editing a skill — a small
     secondary icon/affordance per row (e.g., an "open" icon) rather than removing navigation entirely, so
     founders who want to inspect or edit a skill before using it still can.
3. Do not change `detectExplicitSkillInvocation()` or anything in `api/vcso/chat.ts` — inserting `@slug`
   into the textbox is a UI convenience; the actual routing behavior when the message is sent is already
   correct and independently verified (Phase 3).

### Success criteria
1. Typing in the skills-view search box filters the visible skill list by name/slug/description substring
   match, with no new network request.
2. Clicking a skill row inserts `@{slug} ` into the currently active compose textbox and focuses it,
   without discarding any text the founder had already typed.
3. A founder can still reach the full Skills & Plugins workspace from the ChatRail skills view (view/edit
   path preserved, just no longer the *only* action on a row).
4. No global/private toggle or scope control is introduced anywhere in this UI (standing constraint,
   unchanged from `04-03-PLAN.md`).

## Out of scope for this refinement
- Real Anthropic tool-calling / a tool-execution loop inside Virtual CSO chat where sub-agents can *invoke*
  a skill mid-turn as a callable tool. That is a materially larger change (Phase 7's job — sandbox tool
  integration) and was explicitly separated out in discussion with London: inserting `@slug` into the
  compose box is a UI/routing convenience using Phase 3's already-built explicit-invocation path, not a new
  agentic tool-calling capability. If London wants that larger capability pulled forward ahead of Phase 7,
  that needs its own dedicated planning pass, not a fold-in here.
- Any change to `classify()`, `scoreSkill()`, or `detectExplicitSkillInvocation()` (Phase 3, done and
  independently verified — untouched by this refinement).
- Any change to 04-01's backend CRUD/import/export endpoints (unaffected by either part of this refinement).
