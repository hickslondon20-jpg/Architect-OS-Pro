create/update/delete a `skill_packs` row scoped to the founder's own rows, plus SKILL.md

## Backend surface (Python, `python-backend/main.py`, mirroring the existing `/api/ingest` pattern)

**`POST /api/skills`** — create a private skill for the requesting founder.
Body: `{ user_id, name, description, domain?, skill_kind?, trigger_tags?, required_platform_context?,
body }`. Inserts a `skill_packs` row with `scope='private'`, `user_id` from the authenticated caller
— never accept `scope` in the request body at all (not even to reject it — simplest guarantee: the
field doesn't exist in the request schema).

**`PATCH /api/skills/{id}`** — update a founder's own skill. Reject if the row's `user_id` doesn't
match the caller (defense in depth — RLS already enforces this at the DB layer if using the founder's
own client; if this route uses a service-role client for any reason, this check becomes load-bearing,
not optional).

**`DELETE /api/skills/{id}`** — delete a founder's own skill. Cascades to `skill_files` (FK
`ON DELETE CASCADE` already in place from Phase 1) and should also delete the corresponding
Storage objects under `skill-files/{user_id}/{skill_id}/` — don't leave orphaned blobs.

**`POST /api/skills/import`** — accepts a ZIP upload. Extracts `SKILL.md` (parse via the helper
above), and any `scripts/`, `references/`, `assets/` subdirectories — each file becomes a
`skill_files` row + an upload to `skill-files/{user_id}/{skill_id}/{category}/{filename}`. Creates
the `skill_packs` row (`scope='private'`, owned by the importing founder) in the same transaction/
flow as the file uploads — if any file upload fails partway, don't leave a half-created skill (clean
up or make the whole operation retryable).

**`GET /api/skills/{id}/export`** — assembles a ZIP: `SKILL.md` (via `serialize_skill_md`) at the
root, plus every associated `skill_files` row's blob under its `category/` subdirectory. Streams the
ZIP back; does not persist it anywhere new (no export-history table — out of scope).

## Success Criteria

1. A manual-form-shaped `POST /api/skills` call creates a valid `skill_packs` row with the caller's
   `user_id`, `scope='private'` — always, never accepting a client-supplied `scope`.
2. `PATCH`/`DELETE` only ever succeed against the caller's own rows; verified with a cross-founder
   test (founder B cannot patch/delete founder A's skill), not just RLS assumed to cover it.
3. `parse_skill_md`/`serialize_skill_md` round-trip: export a test skill, re-import the resulting ZIP,
   confirm the re-imported row is equivalent to the original (same name/description/body/extension
   fields) — this is the actual "exports losslessly" success criterion from `ROADMAP.md`.
4. A ZIP with only `name`/`description`/body (no ArchitectOS-extension frontmatter keys — simulating a
   genuinely third-party SKILL.md) imports successfully with sensible defaults, not an error.
5. Importing a ZIP with categorized `scripts/`/`references/`/`assets/` files produces correct
   `skill_files` rows and `skill-files` bucket uploads at the right paths.
6. Deleting a skill removes its Storage objects, not just its DB rows.
7. No route in this plan accepts a client-supplied `scope` value under any field name.
