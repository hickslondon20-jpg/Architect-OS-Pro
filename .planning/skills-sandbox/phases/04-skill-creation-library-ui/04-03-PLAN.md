## Goal

Two UI pieces, confirmed together at the checkpoint:
1. **The fourth intelligence peer** — a new full workspace (working name "Skills & Plugins") at
   `/pro/intelligence/skills`, housing the full Skills Library (browse global + own) and both
   creation surfaces (manual form + the 04-02 AI-guided flow).
2. **A lightweight Chats/Skills toggle inside Virtual CSO's `ChatRail`**, showing a condensed,
   read-only library view, linking out to the full workspace for anything beyond browsing.

## Part A — Skills & Plugins workspace (fourth peer)

**Routing (`App.tsx`, mirroring the `/pro/intelligence` block at lines 243–269):**
```tsx
<Route
  path="skills"
  element={
    <FeatureGate featureKey="skills_library" lockedElement={<SkillsLibraryLocked />}>
      <Pro.SkillsWorkspace />
    </FeatureGate>
  }
/>
```
Add a `SkillsLibraryLocked` component mirroring `VirtualCSOLocked`/`OSEngineLocked`'s existing shape.

**`lib/featureGates.ts`:** add `skills_library` to the `FeatureKey` union, `FEATURE_GATES` (matching
`virtual_cso`/`os_engine`'s `unlockWeek` convention — confirm the right week with London or default
to the same week as the rest of this build's features unless told otherwise), and
`PATH_FEATURE_GATES` (`{ path: '/pro/intelligence/skills', featureKey: 'skills_library' }`).

**`pages/ProSuite/IntelligenceLanding.tsx`:** add a fourth entry to `intelligenceResources` (icon,
label, description, `href: '/pro/intelligence/skills'`, a "when to reach for this" line matching the
existing three's tone). Update the header copy from "Three intelligence peers" / "Three tools, always
in context" to "Four" (or rephrase to avoid the count entirely if that reads more naturally once a
fourth tile exists — use judgment, but don't leave stale "three" copy next to four tiles).

**`SkillsWorkspace` (new page component):**
- Library view: lists global skills + the founder's own private skills. Uses ArchitectOS design
  tokens exactly as `IntelligenceLanding.tsx` does (`Card` from `components/ui`, `aos-h1`/`aos-body`/
  `aos-eyebrow` classes, `--aos-brass`/`--bg-surface` variables) — no default Tailwind grays, no
  Inter, asymmetric layout (not three equal-width cards — see `DESIGN-GUIDE-QUICK.md` anti-patterns).
  No global/private toggle exposed anywhere — scope is never a user-facing control (Phase 1
  amendment).
- Manual creation form: calls `POST /api/skills` (04-01).
- AI-guided flow: mounts the 04-02 conversational component.
- Import: a file-upload control calling `POST /api/skills/import` (04-01).
- Export: a per-skill action calling `GET /api/skills/{id}/export` (04-01), triggering a download.

## Part B — Virtual CSO `ChatRail` Chats/Skills toggle

**`components/pro-suite/virtual-cso/ChatRail.tsx`:** add a small segmented control (e.g. "Chats" |
"Skills") directly above the existing "New chat" button (~line 69). When "Skills" is active, replace
the pinned/recent chat list with a condensed list of the founder's own visible skills (global + own —
same scoping `loadIpLayer()` already does server-side; this needs its own lightweight fetch, e.g. a
`GET /api/skills` list endpoint reusing 04-01's read path, or a direct scoped Supabase query from the
founder's own client using the RLS policies Phase 1 already put in place — either is fine, prefer
whichever is less new surface area). Each row links to `/pro/intelligence/skills` (optionally deep-
linked to that specific skill) — this view is browse-only, no creation/edit/delete affordances here.

## Success Criteria

1. `/pro/intelligence/skills` is reachable, gated the same way as the other three peers, and shows
   both the full library and both creation surfaces without feeling like an afterthought bolted onto
   another tool.
2. `IntelligenceLanding.tsx` no longer says "three" anywhere while showing four tiles.
3. No global/private toggle appears anywhere in the manual creation form or anywhere else in this
   workspace.
4. The Skills Library UI is styled entirely with ArchitectOS design tokens — spot-check against
   `DESIGN-GUIDE-QUICK.md`'s anti-patterns list (no Inter, no pure black, no three-equal-cards, no
   Tailwind default grays).
5. The `ChatRail` Chats/Skills toggle switches views correctly; the condensed skills list shows global
   + the founder's own private skills only (never another founder's private skills); clicking a skill
   navigates to the full workspace.
6. A newly created skill (via any of the three creation paths) is immediately usable in Virtual CSO
   chat via ranked `classify()` or `@slug` — Phase 3's routing needs zero changes to pick it up;
   verify this end to end as part of this plan's testing, not just assumed from Phase 3 being "done."

## Out of Scope

Any change to `api/vcso/chat.ts`, `classify()`, or `detectExplicitSkillInvocation()` (Phase 3, done —
this plan only needs to *verify* a new skill flows through unchanged, not modify that code).
