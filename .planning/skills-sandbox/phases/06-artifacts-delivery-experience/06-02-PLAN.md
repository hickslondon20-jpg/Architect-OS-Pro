# Plan 06-02 — Frontend: Reader Panel Wiring + Delivery Card

**Phase:** 06 — Artifacts & Delivery Experience
**Plan:** 2 of 2
**Depends on:** 06-01 (backend delivery service + verification route must exist and be proven working)
**Decisions:** `CONTEXT.md` §2, §4 (this folder)

---

## Goal

A founder sees a rendered artifact directly in the existing Reader panel (markdown/HTML) or a working
download card inline in chat (everything else) — using the two components identified in research, not
new ones built from scratch for the renderable path.

---

## Pre-Execution Checks

1. Confirm 06-01's verification route (`POST /api/artifacts/verify`) is live and returns both a
   renderable and non-renderable result shape — this plan's frontend work has nothing real to point at
   until that's true.
2. Re-read `components/pro-suite/shared/Reader.tsx` and the `readerPageId`/`readerPage` wiring in
   `VirtualCSOWorkspace.tsx` (around lines 453–467) directly before touching either — confirm the exact
   current prop/state shape hasn't changed since `06-RESEARCH.md` was written.
3. Re-read `components/pro-suite/virtual-cso/AgentStepsPanel.tsx` and `MessageBubble.tsx` for the
   inline-card styling/placement precedent before building the new delivery card.

---

## Part A — Reader panel wiring (renderable artifacts)

Extend the existing `readerPageId`/`readerPage` mechanism in `VirtualCSOWorkspace.tsx` to also resolve
artifact IDs, not just wiki page IDs. Concretely: either (a) namespace the ID (e.g.
`artifact:{uuid}` vs. a bare wiki page ID) and branch the resolution logic on that prefix, or (b) add a
parallel `readerArtifactId` state that also feeds the same `<Reader>` render — pick whichever fits the
existing code's shape better once you're looking at it directly (this is an implementation choice, not a
design one). Either way: **one `<Reader>` render in the JSX tree, not two** — do not duplicate the panel
for artifacts.

Fetch path: a new lightweight API call (e.g. `getArtifact(artifactId)` in `lib/virtualCsoApi.ts` or a
new `lib/artifactsApi.ts`, matching whichever file the existing `virtualCsoApi.ts` functions suggest) that
hits the Python backend for the artifact's renderable content, feeding `Reader`'s `content` prop exactly
as wiki pages already do.

## Part B — Delivery card (non-renderable artifacts)

New component, e.g. `components/pro-suite/virtual-cso/ArtifactDeliveryCard.tsx`, styled per
`AgentStepsPanel.tsx`'s conventions (`rounded-[var(--radius-xs)] border border-[var(--aos-mist)]
bg-[var(--bg-sunken)]`, ArchitectOS design tokens throughout — no Tailwind default grays, no Inter).
Shows filename, a human-readable size, and a download button/link using the signed URL from 06-01's
delivery result. Slots into the same place in a message's render tree that `AgentStepsPanel` currently
occupies (check `MessageBubble.tsx` directly for exactly where) — inline within the relevant assistant
message, not a separate floating element.

## Part C — Wire both paths to a manual trigger for this phase's own verification

Since Phase 7 (not this phase) decides *when* a real chat turn produces an artifact, this phase needs its
own way to prove the UI paths work without that wiring existing yet. Add a small, temporary
dev-only trigger (e.g. a button gated behind a query param or existing dev-tooling pattern already used
elsewhere in this codebase — check for one before inventing a new gating mechanism) that calls 06-01's
verification route and renders whatever comes back through both paths (Reader panel for the markdown
result, delivery card for the non-renderable result). This is scaffolding for this phase's own
verification, not a founder-facing feature — flag clearly in code comments that it's temporary and should
be removed or repurposed once Phase 7 wires the real trigger.

---

## Success Criteria

1. A renderable artifact (from 06-01's verification route) opens in the existing `Reader` panel with
   correctly rendered markdown content — confirmed by actually opening it in a running app, not just by
   the code compiling.
2. A non-renderable artifact renders as an inline delivery card with a working download link that
   actually downloads the correct file when clicked.
3. No new panel component was created — `Reader.tsx` itself is unmodified in its own props/behavior,
   only fed new data.
4. The new delivery card uses ArchitectOS design tokens exclusively (spot-check against
   `DESIGN-GUIDE-QUICK.md`'s hard-no list: no Inter, no pure black, no Tailwind default grays).
5. The temporary dev-trigger scaffolding from Part C is clearly marked as temporary in code comments and
   does not leak into any founder-facing path (e.g. not reachable without the explicit dev gate).

## Out of Scope

Any backend change (06-01 is already done by the time this plan starts). Real chat-turn-triggered artifact
creation (Phase 7) — Part C's manual trigger is scaffolding, not that feature.
