## Goal

A durable, shared place for sandbox-generated files to land, with a schema that's genuinely reusable by
the not-yet-built Domain Agents system (ARTIFACT-04), and a service that can pull a file out of a live
sandbox session and deliver it in whichever shape the caller needs — raw content for rendering, or a
signed URL for download.

## Pre-Execution Checks

1. Confirm no `artifacts` table or bucket already exists (per `06-RESEARCH.md`, none did as of
   2026-07-01 — re-check live via Supabase before assuming this is still true).
2. **Test `copy_from_runtime()` against a real session on the live cluster before writing any other

   path, write a small test file inside it (e.g. `session.run("open('/sandbox/test.txt','w').write('hi')")`),

   mirror it, don't reinvent it.

## Part A — Schema

`docs/migrations/011_artifacts.sql`, modeled directly on `20260701_skill_files_storage.sql`'s
bucket+RLS+trigger shape (simplified — no global-read variant, every artifact is single-owner) and
`009_sub_agent_orchestration.sql`'s `agent_context_sources` polymorphic-source shape:

```sql
-- Bucket
INSERT INTO storage.buckets (id, name, public)
VALUES ('artifacts', 'artifacts', false)
ON CONFLICT (id) DO NOTHING;

-- Metadata table
CREATE TABLE IF NOT EXISTS public.artifacts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL,
  source_kind   TEXT NOT NULL CHECK (source_kind IN ('vcso_thread')),
  source_id     UUID,
  filename      TEXT NOT NULL,
  mime_type     TEXT,
  size          BIGINT,
  storage_path  TEXT NOT NULL,
  renderable    BOOLEAN NOT NULL DEFAULT false,
  description   TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS artifacts_user_id_idx ON public.artifacts (user_id);
CREATE INDEX IF NOT EXISTS artifacts_source_idx ON public.artifacts (source_kind, source_id);

-- updated_at trigger: mirror update_skill_files_updated_at exactly (reuse the function if a
-- generic one already exists in this schema; check before creating a near-duplicate).

ALTER TABLE public.artifacts ENABLE ROW LEVEL SECURITY;
GRANT SELECT, INSERT, UPDATE, DELETE ON public.artifacts TO authenticated;

-- RLS: owner-only, no global variant (unlike skill_files)
CREATE POLICY "artifacts_select_own" ON public.artifacts FOR SELECT
  USING (user_id = (SELECT auth.uid()));
CREATE POLICY "artifacts_insert_own" ON public.artifacts FOR INSERT
  WITH CHECK (user_id = (SELECT auth.uid()));
CREATE POLICY "artifacts_delete_own" ON public.artifacts FOR DELETE
  USING (user_id = (SELECT auth.uid()));

-- Storage RLS: owner-folder pattern, mirroring skill-files/raw-documents exactly
CREATE POLICY "artifacts_select_own_folder" ON storage.objects FOR SELECT
  USING (bucket_id = 'artifacts' AND (storage.foldername(name))[1] = (SELECT auth.uid())::text);
CREATE POLICY "artifacts_insert_own_folder" ON storage.objects FOR INSERT
  WITH CHECK (bucket_id = 'artifacts' AND (storage.foldername(name))[1] = (SELECT auth.uid())::text);
CREATE POLICY "artifacts_delete_own_folder" ON storage.objects FOR DELETE
  USING (bucket_id = 'artifacts' AND (storage.foldername(name))[1] = (SELECT auth.uid())::text);
```

Note the deliberate omission of an `UPDATE` policy on `storage.objects` (files are replaced by
delete+re-insert, not mutated in place — matches the `raw-documents` convention; verify this against the
actual `raw-documents` migration before assuming, don't just copy this plan's guess verbatim). Storage
path convention: `{user_id}/{artifact_id}/{filename}`, matching every existing bucket's
first-path-segment-is-owner convention exactly (this is what the RLS policies above depend on).

## Part B — Delivery service

New `python-backend/services/artifact_service.py`, mirroring `sandbox_service.py`'s class shape:

```python
class ArtifactService:
    def __init__(self, settings: Settings, supabase_client: Client, sandbox_service: SandboxService) -> None: ...

    @classmethod
    def from_env(cls) -> "ArtifactService": ...

    def deliver_from_sandbox(
        self,
        user_id: str,
        thread_id: str,
        container_path: str,
        filename: str,
        description: str | None = None,
    ) -> ArtifactDeliveryResult:
        """
        1. Reuse the existing live session for `thread_id` from `self._sandbox_service` — do not open a
           new session; the file must be extracted from the session that actually produced it.
        2. Extract the file via copy_from_runtime() (or the exec-based fallback per Pre-Execution
           Check #2).
        3. Determine renderability from file extension/mime type — markdown (.md) and HTML (.html) are
           renderable; everything else (pptx, xlsx, csv, docx, png, etc.) is not.
        4. Upload to the `artifacts` bucket at {user_id}/{artifact_id}/{filename}.
        5. Insert the `artifacts` row (source_kind='vcso_thread', source_id=thread_id).
        6. Return ArtifactDeliveryResult: for renderable content, the raw text content (read back after
           upload, or captured before upload — either is fine, don't re-download from Storage
           unnecessarily); for non-renderable, a signed URL (create_signed_url, 3600 seconds / 1 hour,
           per London's confirmed default) plus filename/size.
        """
```

## Part C — Verification route

`POST /api/artifacts/verify` (mirroring `/api/sandbox/verify`'s shape, same `require_ingest_secret`
gating): opens or reuses a test sandbox session, runs code that writes both a markdown file and a
non-renderable file (e.g. a tiny `.xlsx` via `openpyxl`) inside the sandbox, calls
`deliver_from_sandbox` for each, and returns both results — proving the full round trip (sandbox → pod
file → extraction → Storage → `artifacts` row → delivery-shaped response) for both renderability paths
in one call, against the real cluster, not a mock.

## Success Criteria

1. `artifacts` bucket and table exist, RLS confirmed (owner-only, both table and storage objects) —
   verified live via the Supabase MCP, not just by the migration file existing.
2. `copy_from_runtime()` tested directly against the real cluster; if unreliable, the exec-based fallback
   is implemented and proven working instead — either way, a real file produced inside a real sandbox pod
   is successfully extracted to the Python backend.
3. A markdown file produced in a sandbox session is delivered as raw renderable content (verified via the
   verification route, content matches what was written inside the sandbox).
4. A non-renderable file (e.g. `.xlsx`) produced in a sandbox session is uploaded to the `artifacts`
   bucket, and a signed URL is generated that successfully downloads the exact file when fetched — not
   just that a URL string was returned.
5. The `artifacts` row's `source_kind`/`source_id` correctly references the originating thread, and the
   schema has no column or constraint that would prevent a future Domain-Agents `source_kind` value from
   being added additively (i.e., nothing hard-coded assumes `source_kind = 'vcso_thread'` always).
6. Deleting an `artifacts` row also removes its Storage object — no orphaned blobs (mirroring Phase 4's
   skill-files delete behavior).
