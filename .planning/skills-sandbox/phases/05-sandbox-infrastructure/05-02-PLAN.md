## Goal

A container image, built and pushed via Cloud Build (no local Docker daemon anywhere in the chain, per
`CONTEXT.md` §2b), carrying every library the document-generation/calculation use case needs
(SANDBOX-04), plus IPython — required by `InteractiveSandboxSession`'s runner script but not called out
in the original SANDBOX-04 library list, since that list predates the decision to use
`InteractiveSandboxSession` specifically (see `CONTEXT.md` §2c). Referenced by full Artifact Registry path
from 05-03 with `skip_environment_setup=True`.

## Steps

### 1. Author the Dockerfile

Location: `python-backend/sandbox-image/Dockerfile` — deliberately scoped under `python-backend/`, not
repo root, so it's never confusable with a future Railway-service Dockerfile if one is ever added.

```dockerfile
FROM python:3.11-slim

# System packages some of these libraries need at build/runtime (matplotlib font handling,
# lxml used transitively by python-docx/python-pptx)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    libxml2-dev \
    libxslt1-dev \
    && rm -rf /var/lib/apt/lists/*

# Document-generation and calculation libraries (SANDBOX-04), plus IPython — required by
# llm-sandbox's InteractiveSandboxSession runner script (CONTEXT.md §2c), not itself one of
# SANDBOX-04's named libraries but load-bearing for session persistence to work at all.
RUN pip install --no-cache-dir \
    pandas \
    numpy \
    python-docx \
    python-pptx \
    openpyxl \
    matplotlib \
    ipython \
    ipykernel

WORKDIR /sandbox
```

Verify the base image and installed packages satisfy `llm-sandbox`'s "Custom Image Requirements" for
`skip_environment_setup=True` (per `CONTEXT.md` §2c, quoting the library's own docs): "Python interpreter
in expected location (usually `/usr/bin/python` or `/usr/local/bin/python`)" — confirm via `docker run
<image> which python3` equivalent through Cloud Build's build-time verification, or a quick post-push
smoke pod, not just assumed from the base image's reputation.

### 2. Build and push via Cloud Build — no local Docker

```bash
gcloud builds submit \
  --tag us-west2-docker.pkg.dev/architectos-sandbox/sandbox-images/sandbox-python:latest \
  --project=architectos-sandbox \
  python-backend/sandbox-image/
```

Also push a dated tag alongside `:latest` for traceability (so a future rollback doesn't depend on
`:latest` having stayed pointed at a known-good build):

```bash
gcloud builds submit \
  --tag us-west2-docker.pkg.dev/architectos-sandbox/sandbox-images/sandbox-python:2026-07-01 \
  --project=architectos-sandbox \
  python-backend/sandbox-image/
```

(Cloud Build will rebuild rather than re-tag the existing image if run twice — acceptable for this
plan's scope; a smarter build-once-tag-twice approach is a nice-to-have, not a requirement.)

### 3. Verify the pushed image is actually usable

Confirm the image exists in the registry:
```bash
gcloud artifacts docker images list us-west2-docker.pkg.dev/architectos-sandbox/sandbox-images \
  --project=architectos-sandbox
```

Confirm every required library imports cleanly inside the actual pushed image — do not rely on the
Dockerfile's `pip install` step succeeding at build time as sufficient proof. Run a throwaway pod against
the image (using the runtime service account from 05-01) that executes:
```python
import pandas, numpy, docx, pptx, openpyxl, matplotlib, IPython
print("all imports OK")
```
and confirm the output, then delete the throwaway pod. This is the actual SANDBOX-04 success criterion —
"the image includes X, Y, Z" means they import and work, not merely that a `pip install` line mentions
them.

## Success Criteria

1. `python-backend/sandbox-image/Dockerfile` exists, based on a `python:3.11-slim`-class image.
2. Image built and pushed via `gcloud builds submit` (Cloud Build) — no evidence of a local `docker build`
   command anywhere in the process (confirm by checking no Docker daemon was required/available in
   whichever environment ran this plan).
3. `pandas`, `numpy`, `python-docx`, `python-pptx`, `openpyxl`, `matplotlib` all import successfully
   inside a running pod built from the pushed image — verified by actually running a pod, not just by
   the build succeeding.
4. `ipython`/`ipykernel` are present and importable (load-bearing for 05-03's
   `InteractiveSandboxSession` usage, not itself a named SANDBOX-04 library, but required for SANDBOX-04
   to be usable the way this build needs it).
5. The image is tagged both `:latest` and with a dated tag, both present in Artifact Registry.

## Out of Scope

Wiring this image into the Python backend's sandbox service (05-03). Any pod-manifest / security-context
configuration — that's a runtime concern for the `InteractiveSandboxSession` call in 05-03, not something
baked into the image itself.
