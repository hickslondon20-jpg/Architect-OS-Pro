## Part A — Schema

New migration, `docs/migrations/<next-number>_sandbox_sessions.sql` (check the highest existing number in
`docs/migrations/` first — do not guess or reuse a number):

```sql
create table if not exists public.sandbox_sessions (
  id uuid primary key default gen_random_uuid(),
  thread_id uuid not null references public.vcso_chat_threads(id) on delete cascade,
  pod_name text not null,
  kube_namespace text not null default 'default',
  status text not null default 'active' check (status in ('active', 'closed', 'expired')),
  created_at timestamptz not null default now(),
  last_active_at timestamptz not null default now()
);

create index if not exists sandbox_sessions_thread_id_idx on public.sandbox_sessions (thread_id);
create index if not exists sandbox_sessions_status_idx on public.sandbox_sessions (status)
  where status = 'active';

alter table public.sandbox_sessions enable row level security;
-- Service-role only (this table is written/read exclusively by the Python backend's service-role
-- client, never directly by a founder's browser client) — mirror the RLS shape used for other
-- backend-only tables (check an existing backend-only table's policy for the exact precedent
-- before writing this from scratch).
```

Apply and independently verify (row exists, FK resolves, RLS confirmed) the same way every prior phase's
migration was verified — don't just report the migration file was written.

## Part C — Sandbox service

New `python-backend/services/sandbox_service.py`, mirroring `skill_draft_synthesis.py`'s shape (a class,
`from_env()` classmethod, a domain-specific `SandboxServiceError`).

**Authenticating to the GKE cluster from just the JSON key (no `gcloud` CLI available at Railway
runtime):** use `google-auth` to build credentials from the parsed key dict
(`google.oauth2.service_account.Credentials.from_service_account_info`), then `google-cloud-container`'s
`ClusterManagerClient` to fetch the cluster's endpoint + CA certificate
(`get_cluster(name=...)` → `.endpoint`, `.master_auth.cluster_ca_certificate`). Build a
`kubernetes.client.Configuration` with `configuration.host = f"https://{endpoint}"`,
`configuration.ssl_ca_cert` (write the base64-decoded CA cert to a temp file — the `kubernetes` client
library expects a file path here, not raw bytes), and a bearer token from
`credentials.token` (call `credentials.refresh(google.auth.transport.requests.Request())` first to
populate it). **GCP access tokens expire in roughly one hour — the service must refresh this token
whenever it builds a new K8s client, not cache a `Configuration` object indefinitely across the FastAPI
process's lifetime.** This is the exact pattern documented in `llm-sandbox`'s own "Kubernetes Remote
Cluster" configuration section (bearer-token `client.Configuration`, passed as `client=` into
`InteractiveSandboxSession`) — read that section directly before implementing, don't reinvent the
token-refresh mechanics from scratch.

## Part D — Verification route

A narrow, `require_ingest_secret`-gated route (mirroring the existing pattern at `main.py` line 470/487)
— e.g. `POST /api/sandbox/verify` — that: creates or reattaches a session for a test `thread_id`, runs a
first `session.run(...)` that sets a variable, runs a second `session.run(...)` in the same call chain
that reads that variable back, and returns both results. This is the actual mechanism for proving Phase
5's success criteria 2 and 4 live — not a unit test with a mocked K8s client, a real round trip against
the real cluster.

## Success Criteria

1. A single GKE Autopilot cluster (from 05-01) is reachable from the Python backend using only the
   `ARCHITECTOS_GKE_SERVICE_ACCOUNT_KEY` env var — no `gcloud` CLI, no local kubeconfig file.
2. Calling the verification route creates a real pod on the cluster running the 05-02 image, executes
   code in it, and tears it down cleanly on session close — confirmed by checking `kubectl get pods`
   (via the execution agent's own still-authenticated session from 05-01) shows the pod both existing
   during the call and gone afterward.
3. A variable set in one `session.run()` call is readable in a second, later call against the same
   `thread_id` — the actual SANDBOX-02 / Phase 5 success-criterion-4 proof, via the verification route's
   two-call round trip.
4. `sandbox_sessions` rows are created, updated (`last_active_at`), and correctly transition to `expired`
   by the idle-TTL sweep — verified by creating a session, waiting past a shortened test TTL (temporarily
   override `ARCHITECTOS_SANDBOX_IDLE_TTL_MINUTES` for the test, don't wait 20 real minutes), and
   confirming both the DB row and the actual pod are gone.
5. A session whose underlying pod was killed out-of-band (simulate via manual `kubectl delete pod`) is
   detected and recovered from — the service creates a fresh session rather than hanging or erroring
   opaquely on the next call for that `thread_id`.
6. `pandas`/`numpy`/`python-docx`/`python-pptx`/`openpyxl`/`matplotlib` are confirmed importable from
   *inside* a session created through this service (not just 05-02's standalone pod check) — the full
   path (service → cluster → image) works together, not just each piece in isolation.
