## Part A — London's one manual step (Cloud Shell)

Confirmed live this session: project `architectos-sandbox` exists, billing is attached
(`billingAccounts/01E433-5C540A-FA541C`), `gcloud config set project architectos-sandbox` already
succeeded. London runs the following as one block in the same Cloud Shell session:

```bash
PROJECT_ID=architectos-sandbox
REGION=us-west2

# 1. Enable every API this phase needs
gcloud services enable \
  container.googleapis.com \
  compute.googleapis.com \
  artifactregistry.googleapis.com \
  iam.googleapis.com \
  cloudbuild.googleapis.com \
  --project=$PROJECT_ID

# 2. Artifact Registry repo for the sandbox image (05-02 pushes here)
gcloud artifacts repositories create sandbox-images \
  --repository-format=docker \
  --location=$REGION \
  --project=$PROJECT_ID

# 3. Bootstrap service account — temporary, deleted at the end of this phase
gcloud iam service-accounts create phase5-bootstrap \
  --display-name="Phase 5 Bootstrap (temporary — delete after Phase 5)" \
  --project=$PROJECT_ID

BOOTSTRAP_SA="phase5-bootstrap@${PROJECT_ID}.iam.gserviceaccount.com"

# 4. Scoped roles only — not Owner/Editor. If a later gcloud command fails on a
#    permissions error, add the specific missing role rather than escalating broadly.
for ROLE in \
  roles/container.admin \
  roles/artifactregistry.admin \
  roles/iam.serviceAccountAdmin \
  roles/iam.serviceAccountKeyAdmin \
  roles/cloudbuild.builds.editor \
  roles/storage.admin
do
  gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:${BOOTSTRAP_SA}" \
    --role="$ROLE" \
    --condition=None
done

# 5. Generate its key and print it
gcloud iam service-accounts keys create phase5-bootstrap-key.json \
  --iam-account=$BOOTSTRAP_SA

cat phase5-bootstrap-key.json
```

**Handoff:** copy the full JSON output and save it as a file the execution agent's own session can read
— e.g. paste it into a new file at `python-backend/.secrets/phase5-bootstrap-key.json` inside the repo
working directory (create the `.secrets/` folder if it doesn't exist). **Before this file is created,
confirm `.gitignore` excludes `python-backend/.secrets/` — this is the execution agent's very first task
below, done ahead of asking for the key, specifically so the key is never at risk of being committed even
for one commit.**

## Part B — Execution agent's part (scripted, once the bootstrap key is available)

### Pre-Execution Checks
1. **First action, before anything else:** confirm (or add) a `.gitignore` entry for
   `python-backend/.secrets/` in the repo root `.gitignore`. Do not proceed to request or handle the
   bootstrap key until this is confirmed in place.
2. Confirm no `sandbox-images` Artifact Registry repo, no `architectos-sandbox-cluster` GKE cluster, and no
   `sandbox-runtime` service account already exist (idempotency check — this plan should be safely
   re-runnable if it's interrupted partway).

### Steps

1. Activate the bootstrap service account in the execution agent's own shell:
   ```bash
   gcloud auth activate-service-account --key-file=python-backend/.secrets/phase5-bootstrap-key.json
   gcloud config set project architectos-sandbox
   ```
2. Create the GKE Autopilot cluster:
   ```bash
   gcloud container clusters create-auto architectos-sandbox-cluster \
     --project=architectos-sandbox \
     --region=us-west2
   ```
   This takes several minutes. Confirm success via `gcloud container clusters describe
   architectos-sandbox-cluster --region=us-west2 --format="value(status)"` returning `RUNNING` before
   moving on — don't assume the `create-auto` command returning means it's actually ready.
3. Create the **narrower**, permanent runtime service account — this is the one whose key goes into
   Railway, not the bootstrap account:
   ```bash
   gcloud iam service-accounts create sandbox-runtime \
     --display-name="ArchitectOS Sandbox Runtime (Railway)" \
     --project=architectos-sandbox

   RUNTIME_SA="sandbox-runtime@architectos-sandbox.iam.gserviceaccount.com"

   gcloud projects add-iam-policy-binding architectos-sandbox \
     --member="serviceAccount:${RUNTIME_SA}" \
     --role="roles/container.developer" \
     --condition=None

   gcloud projects add-iam-policy-binding architectos-sandbox \
     --member="serviceAccount:${RUNTIME_SA}" \
     --role="roles/artifactregistry.reader" \
     --condition=None

   gcloud iam service-accounts keys create python-backend/.secrets/sandbox-runtime-key.json \
     --iam-account=${RUNTIME_SA}
   ```
   `roles/container.developer` is the minimum needed to create/manage pods against the cluster;
   `roles/artifactregistry.reader` lets the runtime pull the custom image from 05-02. Neither grants
   cluster admin, IAM admin, or billing access — deliberately narrower than the bootstrap account.
4. Confirm cluster reachability end to end: fetch cluster credentials and run a trivial `kubectl` check
   using the **runtime** service account (not the bootstrap one — this proves the actual credential that
   will live in Railway is sufficient, not just the temporarily-elevated bootstrap one):
   ```bash
   gcloud auth activate-service-account --key-file=python-backend/.secrets/sandbox-runtime-key.json
   gcloud container clusters get-credentials architectos-sandbox-cluster --region=us-west2 --project=architectos-sandbox
   kubectl get nodes
   ```
   `kubectl get nodes` returning a node list (Autopilot manages nodes automatically, but the API should
   still report at least one) confirms real reachability with the real runtime credential.
5. Clean up the bootstrap account — it should not persist past this plan:
   ```bash
   gcloud auth activate-service-account --key-file=python-backend/.secrets/phase5-bootstrap-key.json
   gcloud iam service-accounts keys list --iam-account=phase5-bootstrap@architectos-sandbox.iam.gserviceaccount.com
   # delete the specific key ID returned above:
   gcloud iam service-accounts keys delete <KEY_ID> --iam-account=phase5-bootstrap@architectos-sandbox.iam.gserviceaccount.com --quiet
   gcloud iam service-accounts delete phase5-bootstrap@architectos-sandbox.iam.gserviceaccount.com --quiet
   ```
   Also delete the local `python-backend/.secrets/phase5-bootstrap-key.json` file from disk (this agent's
   sandbox restrictions on delete/rename apply to the *connected repo folder* per
   `PYTHON-BACKEND-DEPLOYMENT.md` — if `rm` fails here the same way it did during the deployment saga,
   flag it and ask London to delete that one file herself; don't treat it as blocking the rest of the
   plan).
6. Report the runtime service account key's contents back for London to add as a new Railway secret env
   var (name TBD in 05-03 — likely `ARCHITECTOS_GKE_SERVICE_ACCOUNT_KEY`, holding the raw JSON) — do not
   add it to Railway yourself; you don't have Railway access, same as the Python backend deployment saga.

## Success Criteria

1. `architectos-sandbox-cluster` exists in `us-west2`, status `RUNNING`, confirmed via
   `gcloud container clusters describe`, not just the create command's exit code.
2. `sandbox-images` Artifact Registry Docker repo exists in `us-west2`.
3. A `sandbox-runtime` service account exists with exactly `roles/container.developer` and
   `roles/artifactregistry.reader` — verified via `gcloud projects get-iam-policy architectos-sandbox
   --flatten="bindings[].members" --filter="bindings.members:sandbox-runtime*"`, not assumed from the
   commands having been run.
4. `kubectl get nodes` succeeds using the **runtime** service account's own credentials (not the bootstrap
   account's), proving the credential that will actually live in Railway is sufficient on its own.
5. The bootstrap service account and its key are deleted; the local bootstrap key file is deleted or
   flagged to London for manual deletion if the sandbox's delete restriction blocks it.
6. `.gitignore` excludes `python-backend/.secrets/` — confirmed by checking no `.secrets/` files appear in
   `git status`.

## Out of Scope

Building/pushing the sandbox image itself (05-02). Any Python backend code changes (05-03). Adding the
runtime key to Railway (London's step, reported at the end of this plan, not executed here).
