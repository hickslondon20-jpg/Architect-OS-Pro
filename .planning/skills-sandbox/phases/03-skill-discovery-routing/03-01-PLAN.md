## Change 2 — Scope the skills query to global + own

```ts
const loadIpLayer = async (service: SupabaseClient, allowDraftIp: boolean, userId: string) => {
  const statusFilter = allowDraftIp ? ['draft', 'active'] : ['active'];
  const [rules, prompts, skills] = await Promise.all([
    service.from('ip_rules')...,       // unchanged
    service.from('ip_prompts')...,     // unchanged
    service
      .from('skill_packs')
      .select('id,slug,name,description,domain,skill_kind,trigger_tags,required_platform_context,status,scope,user_id')
      .in('status', statusFilter)
      .or(`scope.eq.global,user_id.eq.${userId}`)
      .order('slug', { ascending: true }),
  ]);
  // ... unchanged error handling / return
};
```
Confirm the exact PostgREST `.or()` filter syntax against the Supabase JS client version in this repo
during implementation — use the client's parameterized filter builder, do not string-concatenate
`userId` into a raw filter string. Update the call site (`loadIpLayer(service, allowDraftIp)` →
`loadIpLayer(service, allowDraftIp, userId)`) — `userId` is already in scope at that call site.

Add `scope`/`user_id` to the `SkillIndexRow` TypeScript type.

`loadSelectedSkillBodies()` needs no additional filter beyond the table-name fix — it only ever

## Change 3 — Explicit invocation via `@slug`

```ts
const detectExplicitSkillInvocation = (text: string, skills: SkillIndexRow[]): SkillIndexRow | null => {
  const mention = text.match(/@([a-z0-9-]+)/i);
  if (!mention) return null;
  const slug = mention[1].toLowerCase();
  return skills.find((skill) => skill.slug.toLowerCase() === slug) ?? null;
};
```

Call this before `classify()` at the point where `route` is computed (currently `const route =
classify(text, ipLayer.skills);`). If it returns a skill, construct the route result directly from
that skill (skip `scoreSkill()`/ranking entirely) — `selected: [skill]`, `primary: skill`,
`required: skill.required_platform_context ?? []`, `confidence: 1`, `reason:` something like
`` `Explicit invocation: @${skill.slug}` ``. If it returns `null`, call `classify(text, ipLayer.skills)`
exactly as today.

**Security note to verify, not just assume:** because `detectExplicitSkillInvocation` searches
`ipLayer.skills` — the same list Change 2 already scoped to global + the requesting founder's own
rows — a founder can never explicitly invoke another founder's private skill by slug, even if they
know or guess it exactly. Confirm this with a real test (see Success Criteria).

## Success Criteria

1. **Live bug fixed and independently verified:** a Virtual CSO chat message completes successfully
   post-fix, where it would have thrown pre-fix. Confirm via a direct test call, not just by reading
   the diff.
2. `classify()` scores global skills and the requesting founder's own private skills together in one
   ranked pass (per `ROADMAP.md` Phase 3 success criterion #1) — verify by creating a test private
   skill row (direct SQL insert is fine) for a test founder and confirming it appears in that
   founder's `ipLayer.skills` and can be matched/ranked.
3. A private skill never appears in another founder's ranked results (success criterion #2) — verify
   with a second test founder's JWT/service-role-simulated call; the first founder's private skill
   must not appear.
4. A message containing `@<slug>` for a skill in the founder's own scoped list loads that skill
   directly regardless of score (success criterion #3) — test with a low-relevance message text that
   would score 0 under normal `classify()` but still resolves via the `@mention`.
5. A message containing `@<slug>` for a skill the founder cannot see (another founder's private skill,
   or a typo) falls through to normal ranked `classify()` behavior — does not error, does not leak
   whether that slug exists.
6. Existing global-skill routing behavior is unchanged for founders with no private skills (success
   criterion #4) — the 6 existing global skills route exactly as they did before this plan, for a
   message with no `@mention`.
7. No changes to `scoreSkill()`'s scoring heuristics themselves.
