# Plan 05-02 — Credential Model (Supabase Vault) + "Coming Soon" Surface

**Requirements:** MCP-02, MCP-03, MCP-04
**Depends on:** 05-01 (MCP registry source + manager). Independent of live connectors (there are none).
**Goal:** A per-user, Vault-backed credential model for MCP connections with the OAuth lifecycle stubbed, plus a read-only "coming soon" connectors surface in the existing Skills & Plugins workspace.

> Read `05-RESEARCH.md`. Supabase **Vault is available** (`vault` schema exists). Secrets go in Vault; `mcp_connections` holds metadata + a Vault reference only. Zero live connectors (L7).

---

## Step 1 — `mcp_connections` table (metadata only, migration)

Create a migration (follow the repo's current convention — check the latest migration number/name). Table `public.mcp_connections`:
- `id` uuid pk, `user_id` uuid not null, `server_name` text, `transport` text (`stdio`/`http`), `config` jsonb (non-secret connection config), `auth_type` text (`api_key`/`oauth2`), `vault_secret_id` uuid null (reference into Vault), `status` text (`coming_soon`/`disabled`/`connected`, default `coming_soon`), `created_at`/`updated_at`.
- **No secret material in this table.**
- **RLS: owner-only** (`user_id = auth.uid()`) for founder read; **service-role** for backend writes/reads. Mirror the RLS posture of existing per-user tables (`skill_packs`, `sandbox_sessions`).

## Step 2 — Vault-backed secret storage (service-role only)

- Store the actual API key / OAuth tokens via **Supabase Vault** (`vault.create_secret(secret, name, description)`), keeping the returned secret id in `mcp_connections.vault_secret_id`.
- Read/decrypt only via `vault.decrypted_secrets` on the **service-role side**, at the moment the backend makes an MCP call. Never expose the secret via any public API, even encrypted; never send it to the browser.
- Add a small server-side helper (e.g. in `mcp_client.py` or a `mcp_credentials.py`) that, given a connection, fetches its decrypted secret service-side for the manager. At beta this is exercised only by tests (no connected servers).

## Step 3 — OAuth lifecycle: stub, don't implement (MCP-03)

- Include the *fields* for OAuth (refresh token stored in Vault, `expires_at` in config/metadata) so the shape is forward-compatible.
- **Stub** refresh/rotation/revocation: a documented no-op / `raise NotImplementedError("OAuth lifecycle is scaffolded; live connectors are v2")` path, plus a code comment + COMPLETION note explaining this is deliberate (the reason MCP is scaffold-only). **Do not build a token-refresh scheduler.**

## Step 4 — "Coming soon" connectors surface (MCP-04)

- Backend: expose a read-only endpoint (or reuse the skills/plugins surface's data path) returning the **candidate connector list** (config-driven: QuickBooks/GHL/Notion/CRM) with `status='coming_soon'`. No connect action.
- Frontend: a read-only **connectors section in the existing Skills & Plugins workspace** (`/pro/intelligence/skills`, the Ep4 fourth intelligence peer) listing the candidates as "coming soon," styled in the ArchitectOS design system (Obsidian/Brass/Parchment, no Inter/default grays). **No connect button, no credential entry UI** at beta.
- Keep it minimal — a static-ish, config-driven list. Do not build connection management UI.

## Step 5 — Verify

- Live-apply the migration; verify `mcp_connections` RLS (owner-only read; service-role access; no anon/authenticated secret exposure) with rollback-only rows.
- Vault write/read round-trip works service-side (store a temp secret, read it back via `decrypted_secrets`, delete it) — proving the path without a live connector.
- The "coming soon" surface renders the candidate list read-only; no connect affordance exists.
- Confirm no secret is ever returned by any authenticated (non-service-role) query or API path.

## Success criteria (this plan)

1. `mcp_connections` exists, per-user RLS owner-only + service-role, **no secrets stored in it**; migration applied and verified.
2. Secrets are stored in and decrypted from Supabase Vault, service-role only; never in the browser or any public API response (verified with rollback-only data).
3. OAuth lifecycle is stubbed (fields present, refresh/rotation/revocation not implemented) and documented as intentional; no refresh scheduler built.
4. A read-only "coming soon" connectors section exists in the Skills & Plugins workspace with zero connect/credential UI; candidate list is config-driven.
5. `python -m compileall python-backend` clean; `npm run build`/focused TS check clean for the surface; live migration + Vault round-trip verified (or gaps flagged if service-role/Vault access unavailable in the checkout).
