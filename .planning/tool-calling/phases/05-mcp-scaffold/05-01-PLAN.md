# Plan 05-01 — MCP as a Registry Source Type + Client/Discovery Scaffold

**Requirements:** MCP-01
**Depends on:** Phase 2 (`tool_registry` — the `mcp` discriminator + registration/scope patterns already exist), Phase 4 (read/compute registry tools are bridge-callable — MCP read tools inherit this free).
**Goal:** MCP is a first-class registry source type: a client manager + discovery machinery that registers discovered MCP tools into the *existing* registry as deferred `source="mcp"` citizens — built real, tested against a fake server, and shipped with **zero servers configured**.

> Read `05-RESEARCH.md` first. The registry already carries `ToolSource="mcp"`/`ExecutorKind="mcp"` and a placeholder executor branch. This fills it in without a registry redesign. No live connectors (L7).

---

## Step 1 — `MCPClientManager` (discovery machinery)

Add `python-backend/services/mcp_client.py`:
- Reads configured MCP connections (from `mcp_connections` where `status='connected'` — **none at beta**). For each, connect via the appropriate transport (stdio/http) using the `mcp` Python SDK and call `list_tools()`.
- Convert each discovered tool's `inputSchema` (JSON Schema) → the registry's neutral `json_schema` (near-1:1) and build a `ToolDefinition` (`source="mcp"`, `executor_kind="mcp"`, `loading="deferred"`, `keywords` from the tool name/description, `citation` marking MCP provenance).
- `call_tool(server_name, tool_name, args)` — routes an MCP tool call through the SDK. **At beta this path is never exercised live** (zero connected servers); it is exercised only by tests (Step 4).
- Resilience shape (documented, minimal): on connect/list failure, log and mark the connection unavailable; do not crash startup. Reconnect/backoff is a documented stub, not a scheduler.

**Dependency:** add `mcp` to `requirements.txt` (pinned). Confirm it does not destabilize the Railway build (it has no runtime server to reach at beta — it's for discovery machinery + tests). If a build risk surfaces, fall back to a thin stubbed client interface and flag at checkpoint.

## Step 2 — `register_mcp_tools()` on the registry

- Add a registration path mirroring `register_skill_pack_tools()`: given the manager's discovered tools, `register(...)` each as a deferred `source="mcp"` entry.
- Fill the existing `mcp` executor branch to route through `MCPClientManager.call_tool(...)`, wrapping the result in the Phase 2 **citation envelope** (MCP provenance: `source_kind='mcp'`, server + tool identity).
- **Scope stays D1-neutral:** MCP tools carry `surface_tags`/`capability_hints` (for `RegistryNativeScopeSource`) and, where capability-backed scoping is used, are added to the relevant `allowed_tools`. Do not hardcode; use the swappable resolver.
- **Bridge-callability is free:** a read/compute MCP tool registered this way is automatically callable from Code Mode (Phase 4) because it is a registry citizen. Write-capable MCP tools are **not** exposed to the bridge and get one-writer/authorization treatment later (out of scope at beta).

## Step 3 — Zero servers configured; "coming soon" as the beta state

- Ship with **no configured/connected servers**. `MCPClientManager` initializes, finds zero connections, registers zero MCP tools — the registry, `tool_search`, and catalog behave exactly as Phase 2–4 with no MCP entries.
- The candidate connector list (QuickBooks/GHL/Notion/CRM) is config-driven metadata for the "coming soon" surface (05-02), not connected servers.

## Step 4 — Test against a fake MCP server

- Add a test that stands up a **fake/local MCP server** (or a fake client), exercises discovery (`list_tools` → registry registration), a `call_tool` round-trip returning a citation-enveloped result, and confirms an MCP tool is discoverable via `tool_search` and scoped correctly. This proves the machinery is real without any live connector.
- Confirm the zero-servers production path registers nothing and does not error.

## Success criteria (this plan)

1. `MCPClientManager` connects (transport), discovers (`list_tools`), and converts MCP tool schemas to the registry's neutral schema.
2. `register_mcp_tools()` registers discovered tools as deferred `source="mcp"` registry citizens; the `mcp` executor routes through the manager and returns citation-enveloped results.
3. MCP tools are discoverable via `tool_search` and scoped via the swappable resolver (D1 open); read/compute MCP tools are bridge-callable (Phase 4) with no extra wiring.
4. Zero servers are configured in production; the no-MCP path is unchanged from Phase 2–4.
5. Fake-server test proves discovery + call + scoping; `python -m compileall python-backend` clean; the `mcp` dependency (if added) does not break the build.
