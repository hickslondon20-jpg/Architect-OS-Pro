# Plan 04-01 — Exec-Channel Bridge Core

**Requirements:** BRIDGE-01, BRIDGE-02, BRIDGE-03, BRIDGE-04, BRIDGE-05
**Depends on:** Phase 2 (`tool_registry` + scoped catalog + citation envelopes), Ep4 sandbox (`SandboxService` / `KubernetesInteractiveSandboxSession`). Inherits, does not re-touch.
**Goal:** LLM-generated code in the sandbox can call registry tools via a file-based, exec-channel bridge — the host fulfils each call host-side with real credentials and returns the result into the pod — with the pod holding zero credentials and zero network egress.

> Read `04-RESEARCH.md` and this phase's `CONTEXT.md` first. This is Option B (exec-channel), not the reference's HTTP bridge. Reuse the existing `_write_remote_command`/`_read_remote_result` exec mechanism as the model.

---

## Step 1 — In-pod bridge protocol (files, stdlib only)

Define a file protocol under the sandbox workdir (e.g. `/sandbox/.bridge/`):
- `requests/{request_id}.json` — written by the in-pod `tool_client` stub: `{id, tool_name, arguments}`.
- `responses/{request_id}.json` — written by the host fulfiller: `{id, ok, result | error}`.
- `done.sentinel` — written by the runner when the user cell finishes.

The **in-pod `tool_client`** (stdlib only — no network, no credentials):
- `tool_client.call(tool_name, **kwargs)` writes `requests/{uuid}.json`, then **polls** `responses/{uuid}.json` (short sleep) until present, reads it, returns `result` or raises on `error`.
- Ship it as a pre-written module + per-session **typed stubs** (Step 4) injected at execution start.

## Step 2 — Host-side fulfiller (registry dispatch with credentials)

Add a host-side fulfiller that, given a session's **scoped catalog**, resolves and executes a requested tool:
- Validate `tool_name` is in the session's scoped catalog (Step 3). Reject anything else with a structured error (authorization boundary — BRIDGE-02).
- Execute the tool via its **registry executor** (Phase 2), host-side, with the real service-role Supabase client / API keys. Wrap the result in the Phase 2 **citation envelope** (BRIDGE-05).
- Return JSON to be written into `responses/{id}.json`. Never raise into the pod uncaught — return a structured error dict.

## Step 3 — Concurrency: run the cell while fulfilling tool calls

Extend `KubernetesInteractiveSandboxSession` (already a custom subclass) with a `run_with_bridge(code, fulfiller, scoped_catalog)` method:
1. Inject the `tool_client` + typed stubs (Step 4) into the pod.
2. Launch the user cell as a **background process** in the pod (writes stdout/stderr to files and `done.sentinel` on completion) — so the host is not blocked on `session.run`.
3. **Host poll loop** over the exec channel: `ls requests/` → for each new request, read it, call `fulfiller(...)`, write `responses/{id}.json`; check for `done.sentinel`; short sleep; repeat until done (or timeout / max tool calls).
4. Return final stdout/stderr/exit_code.

Keep the existing synchronous `execute_code` path unchanged for cells that use no tools (bridge is additive, zero-overhead when unused — Success Criterion 6).

## Step 4 — Typed stubs generated from the scoped catalog

For each tool in the session's scoped catalog, generate a typed Python function stub wrapping `tool_client.call(...)` with the right parameter names/types (from the registry's neutral JSON schema), plus a short docstring from the tool description. Inject the generated module into the pod at execution start (stdlib `urllib`-free — file protocol only). This gives the LLM IDE-quality hints (the reference's "typed stubs," over files instead of HTTP).

## Step 5 — Zero network egress

Apply a **deny-all egress NetworkPolicy** to sandbox pods (they need no network under Option B). Confirm GKE Autopilot honors it at the namespace/pod level; do not allow DNS unless proven necessary. Document the policy as infra alongside the cluster config. (Credentials-out is already guaranteed by Option B; this closes exfiltration-out.)

## Step 6 — Standalone harness (test independent of the VCSO loop)

Add a test/harness that: opens a sandbox session, builds a small scoped catalog (e.g. `kb_grep`, `kb_read`), runs code that calls `tool_client.call("kb_grep", pattern="revenue")` then `kb_read(...)`, and asserts the results returned in-pod carry the citation envelope. Prove: (a) a tool in-catalog resolves; (b) a tool out-of-catalog is rejected; (c) `execute_code` is not callable from code; (d) a no-tool cell behaves as before.

## Success criteria (this plan)

1. In-pod `tool_client` + typed stubs call registry tools over the file protocol; the host fulfiller executes them host-side and returns results — one execution, multiple tool calls, no per-tool inference round-trip.
2. The scoped catalog is enforced host-side: out-of-catalog tools, `execute_code`, and KB-write tools are rejected (BRIDGE-02).
3. The pod holds no credentials and has zero network egress; sandbox code sees only results (BRIDGE-03/04).
4. Results carry the citation envelope (BRIDGE-05).
5. Plain `execute_code` with no tool calls is unchanged (zero-overhead bridge).
6. Harness passes; `python -m compileall python-backend` clean. Live GKE smoke run or flagged if creds unavailable.
