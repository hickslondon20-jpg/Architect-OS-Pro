# Plan 06-01 — Per-Thread Degradation % Signal

**Requirements:** DEGRADE-01
**Depends on:** Phase 1 (`ai_usage_log` `role='main'`), Phase 3 (the VCSO loop + SSE stream). Inherits, does not re-touch.
**Goal:** A per-thread "% remaining before degradation" signal, computed from the current thread's peak `role='main'` prompt tokens vs. the model's context window, streamed to the browser and rendered as a slim % bar — percentage, never raw tokens.

> Read `06-RESEARCH.md` first. The main window fills from the assembled per-turn context + in-turn tool-result accumulation (not a growing append-list); the right measure is the turn's peak `role='main'` `input_tokens`. `ai_models` has no window column yet.

---

## Step 1 — Put the model context window in the registry

Add `context_window integer` to `public.ai_models` (migration; follow the repo's current naming convention). Seed the window for the active Claude model (`claude-sonnet-4-6`). Read it via the model registry alongside `resolve_platform_model` (so the value is registry-driven and a future admin dropdown can adjust it), with a config default (e.g. `ARCHITECTOS_LLM_CONTEXT_WINDOW`) as the fallback when a row/value is missing.

## Step 2 — Compute the per-thread degradation %

- The measure is the **peak `role='main'` `input_tokens` for the latest turn** in the thread ÷ the resolved `context_window`. `vcso_chat_service` already captures per-round `input_tokens` and `final_usage_input`; use the max (the largest prompt the model saw that turn), which is the most accurate fullness measure.
- Compute **% remaining** = `1 - (peak_main_input_tokens / context_window)`, clamped to [0,1].
- Read only `role='main'` events for the thread (exclude `sub_agent`/`utility` — L13). Sub-agent internal usage never counts toward thread fullness.

## Step 3 — Stream the signal

- Emit the signal from `vcso_chat_service` as a small dedicated SSE event (e.g. `usage` / `context`) — lean over extending `done`, so it can update independently. Payload carries the **percentage** (+ a coarse band: green/amber/red), **not** raw tokens.
- Keep it per-turn: emit after the turn's peak prompt is known (end of loop / on `done`).

## Step 4 — Render the % bar (frontend)

- A slim bar in the chat input footer (within the existing `max-w-3xl` container), showing **% remaining** with a color band: green (≥ ~40% remaining) → amber (~20–40%) → red (< ~20%). Text label is a percentage (e.g. "72% context remaining"), never raw tokens or cost.
- Per-thread; **resets on thread switch**; appears once a turn has produced a usage signal.
- ArchitectOS design tokens (Obsidian/Brass/Parchment, Geist/Geist Mono, no Inter/default grays).
- Extend the existing frontend SSE handling (`virtualCsoApi.ts` `parseSseStream` is generic) for the new event; add an `onUsage`/`onContext` callback and thread-scoped state in the workspace.

## Step 5 — Verify

- For a thread with a heavy tool-use turn, the % drops as expected and reflects peak main `input_tokens` / window; a light turn shows high remaining.
- Sub-agent-heavy turns do **not** deflate the signal beyond what the main window actually holds (L13 exclusion proven).
- The bar shows a percentage only; resets on thread switch; renders in design tokens with no regression.

## Success criteria (this plan)

1. `ai_models.context_window` exists and is read via the registry (config fallback); the active Claude window is seeded.
2. The per-thread % is computed from peak `role='main'` `input_tokens` ÷ window, `role='main'` only (sub-agent/utility excluded).
3. The signal streams as a dedicated SSE event carrying a percentage (+ band), not raw tokens.
4. A slim % -remaining bar renders per-thread in the chat footer, color-banded, resets on thread switch, design-token styled.
5. `python -m compileall python-backend` + `npm run build` clean; behavior verified live or gaps flagged.
