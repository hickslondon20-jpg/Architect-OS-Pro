# Plan 06-02 — Context Compaction

**Requirements:** DEGRADE-02
**Depends on:** 06-01 (the degradation signal / threshold), Phase 3 (`_build_context`). Inherits Phase 1 usage tagging.
**Goal:** Condense a long thread's assembled context so the freshly-built prompt stays under the window and the thread can keep going — reclaiming context, not refunding cost.

> Read `06-RESEARCH.md` and this phase's `CONTEXT.md`. The VCSO re-assembles bounded context per turn, so compaction targets the *assembled context* (prior tool results + older thread content), not a growing append-list. Compaction is not writeback (`/api/vcso/writeback` → OS Engine is a different job).

---

## Step 1 — Compaction service (utility-model summarization)

Add a compaction service that, given a thread, summarizes the older/heavier portion of its assembled context — primarily `_load_prior_tool_results` output and older recent-message contents — into a compact running summary.
- Run the summarization on the **Python direct-Anthropic lane** via the model-settings registry (a utility/cheaper model is appropriate). **Tag the usage `role='utility'`** (Phase 1). This is why compaction reclaims context but does not refund cost (L13).
- Keep the summary faithful and lossy-by-design: preserve decisions, figures, and commitments; drop verbose intermediate tool output.

## Step 2 — Store the compacted summary (reuse, minimal)

- Store a per-thread compacted summary minimally — lean: a column/JSONB on `vcso_chat_threads` (e.g. `compacted_summary`, `compacted_through_message_id`, `compacted_at`) — rather than a new subsystem. Confirm the minimal shape at build-planning.

## Step 3 — Consume the summary in `_build_context`

- Extend `_build_context` so that, when a compacted summary exists, it uses the summary in place of the compacted-away prior tool results / older content, keeping the assembled prompt lean while preserving continuity.
- Newer turns after the compaction point remain full-fidelity; only the compacted range is summarized.

## Step 4 — Trigger

- Lean: **founder-initiated** first — a "compact this thread" affordance that appears when the 06-01 signal enters the amber/red band; optionally an auto-trigger at a threshold as a follow-on.
- After compaction, the 06-01 signal recomputes higher (context reclaimed) on the next turn.

## Step 5 — Verify

- A long, tool-heavy thread near the threshold: after compaction, the next turn's assembled prompt is materially smaller and the degradation % recovers; the conversation continues coherently (decisions/figures preserved).
- The compaction call emits a `role='utility'` usage event; prior turns' metered spend is unchanged (cost not refunded).
- Compaction does not write to the knowledge base and is distinct from writeback.

## Success criteria (this plan)

1. A compaction service summarizes the older/heavier assembled context on the Python direct-Anthropic lane, tagged `role='utility'`.
2. The compacted summary is stored minimally (thread column/JSONB) and consumed by `_build_context`, keeping newer turns full-fidelity.
3. Post-compaction, the assembled prompt is materially leaner and the degradation % recovers; the thread continues coherently.
4. Compaction reclaims context but does not refund cost (verified: utility-tagged summarization call; prior spend unchanged); it is not writeback.
5. `python -m compileall python-backend` + `npm run build` clean; behavior verified live or gaps flagged.
