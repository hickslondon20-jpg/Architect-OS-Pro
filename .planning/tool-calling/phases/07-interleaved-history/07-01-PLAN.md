# Plan 07-01 — Persistence Completeness + Carry the Rich Trace Through Reconstruction

**Requirements:** HIST-03 (persistence half), HIST-01 (data-carry half)
**Depends on:** Phase 3 (persisted trace via `agent_delegation_runs`/`steps`), Phase 2 (citation envelopes). Inherits, does not re-touch the loop core.
**Goal:** Ensure the persisted trace is citation-ready (`source_refs` populated) and that the rich fields already in the schema (`step_type`, `title`, `summary`, `source_refs`) are carried through reconstruction to the client — the data foundation the 07-02 rendering consumes.

> Read `07-RESEARCH.md` first. The schema already has `step_type` + `source_refs` + `title`/`summary`/`metadata`; the client `toAgentStep` flattens steps to `{tool, input, output, status}` and may not persist `source_refs`. No migration needed — the columns exist.

---

## Step 1 — Verify + wire `source_refs` population (backend)

- Confirm whether `vcso_chat_service` (and the sub-agent services — KB Explorer, sandbox) write the Phase 2 citation envelope (`source_kind`/`source_id`/`verbatim`/label) into `agent_delegation_steps.source_refs` when persisting steps.
- If not populated, **wire it:** when a tool result carries a citation envelope, write its `sources` into the step's `source_refs` column. Keep it forward-compatible with the four-tier citation taxonomy (Tier 0 records, Tier 1 wiki, Tier 2/3 docs, web).
- Also confirm `step_type` is set meaningfully per step (tool call vs `delegate_to_sub_agent` sub-agent run vs code-execution vs `context_build`). If steps are all one generic type, set `step_type` so the renderer can differentiate. No migration — the columns exist.

## Step 2 — Carry the rich fields through reconstruction (frontend)

- Extend the `AgentStep` type and `toAgentStep` in `lib/virtualCsoApi.ts` to carry `step_type`, `title`, `summary`, and `source_refs` (in addition to the current `tool`/`input`/`output`/`status`).
- Update `getMessagesForChat`'s `agent_delegation_steps` select to include `step_type`, `title`, `summary`, `source_refs` (they're already columns).
- Do the same for the **live** path: the Phase 3 SSE `step`/`tool_call`/`tool_result` events should carry `step_type` + `source_refs` so live and reload feed the same enriched shape (07-02 renders both identically).

## Step 3 — Keep it curated (L11)

- `source_refs` carries source identity + verbatim spans for the future citations UI, but the trace rendering (07-02) stays curated — do not expand `input`/`output` into raw payloads on the client. `input_summary`/`output_summary` remain the safe summaries.

## Step 4 — Verify

- A reloaded thread's steps now carry `step_type` + `source_refs` to the client (inspect the reconstructed `agentSteps`).
- For a KB-read-bearing turn, `source_refs` contains the document source identity + verbatim (populated in persistence); for `execute_code`, computation provenance; for a step with no source, an empty `source_refs`.
- Live and reload produce the same enriched step shape.

## Success criteria (this plan)

1. `agent_delegation_steps.source_refs` is populated from the Phase 2 citation envelope (verified; wired if it wasn't); `step_type` is set meaningfully per step. No migration.
2. `toAgentStep` / `getMessagesForChat` carry `step_type`, `title`, `summary`, `source_refs` to the client; the live SSE path carries the same.
3. The rendering (07-02) has the data it needs for typed panels + future citations; trace stays curated (no raw payloads).
4. `python -m compileall python-backend` + `npm run build` clean; reconstruction verified live or gaps flagged.
