# Plan 02-01 — Registry Core, Sources, Adapters, and `tool_search`

**Requirements:** REG-01, REG-02, REG-03
**Depends on:** Phase 1 (live model routing + `usage_events.py`). Inherits, does not re-touch.
**Goal:** A single in-process tool registry that holds tool definitions in a neutral schema with a `source` discriminator, is populated from native code + `skill_packs`, adapts to the Anthropic (hot path) and OpenAI (utility) tool formats, and exposes a `tool_search` retrieval function + meta-tool.

> Read `02-RESEARCH.md` first. There is no existing registry; tools are hardcoded Anthropic-format lists per sub-agent; skills load-by-body. This is a Python module, not a migration.

---

## Step 1 — `ToolDefinition` and the neutral schema

Create `python-backend/services/tool_registry.py` (name per repo convention). Define a `ToolDefinition`:

- `name: str` (unique key)
- `description: str` (one-liner used in the compact catalog)
- `json_schema: dict` — the parameters as **plain JSON Schema** (the shared core; identical payload whether wrapped as Anthropic `input_schema` or OpenAI `parameters`)
- `source: Literal["native", "skill", "mcp"]`
- `executor` — an abstraction covering the two live executor kinds (+ a future MCP kind):
  - `native`: a callable `(user_id, thread_id, tool_input, …) -> result_envelope`
  - `skill`: not a callable — a resolver that returns the skill's `body` (+ metadata) to inject into context ("load-by-body"); represent this explicitly, do not force it into a function signature
  - `mcp`: placeholder (Phase 5) — a marker that routing goes through an MCP client
- `loading: Literal["always", "deferred"]` — native core tools `always`; skills and MCP `deferred` (discovered via `tool_search`)
- `citation` — the result-envelope descriptor (see 02-02): whether/how this tool's output carries source identity + verbatim text
- optional `capability_hint` / surface tags — see 02-02's D1-neutral resolver (kept injectable, not authoritative here)

## Step 2 — Format adapters (Anthropic is the hot path)

Provide two pure functions on the registry:

- `to_anthropic(defs) -> list[{"name","description","input_schema"}]` — **the canonical output for the live Claude loops.**
- `to_openai(defs) -> list[{"type":"function","function":{"name","description","parameters"}}]` — for utility-model interop only.

Both wrap the same `json_schema`. Keep this thin — the near-identity of the two formats is the point; do not build a schema-translation layer beyond the envelope.

## Step 3 — Register native tools (fold in what exists)

Register the existing native tools into the registry as `source="native"`, `loading="always"`, moving each tool's definition next to its executor:

- KB Explorer's 8 tools (`kb_ls/kb_tree/kb_grep/kb_glob/kb_read/wiki_search/wiki_get_page/wiki_list`) — schemas lifted verbatim from `KB_EXPLORER_TOOLS`, executors pointing at the existing `KbNavigationService`/`DocWikiReadService` dispatch.
- Sandbox's 2 tools (`execute_code`, `read_skill_file`) — from `SANDBOX_EXECUTION_TOOLS`, executors pointing at the existing sandbox/skill-file dispatch.

**Verbatim schemas, identical behavior.** This is a re-home, not a redesign. (The actual rewiring of the services to *source from* the registry is 02-02, behavior-preserving.)

## Step 4 — Skill source adapter (`skill_packs` → deferred registry entries)

Add a function that reads `skill_packs` (respecting the same global-or-owner visibility the app already enforces — global scope OR `user_id == requester`) and yields `ToolDefinition`s with `source="skill"`, `loading="deferred"`:

- `name` ← skill `slug`; `description` ← skill `description`; catalog keywords ← `trigger_tags`.
- `json_schema` — skills take no structured parameters; use a minimal/empty schema (a skill is "loaded," not "called with args"). If a nominal arg is needed for uniform tool-calling, keep it trivial (e.g. none required).
- `executor` = the skill "load-by-body" kind: returns the skill `body` (+ `output_contract`/`writeback_rules` as today) for injection — mirroring Ep4's existing `loadSelectedSkillBodies` semantics, not a new mechanism.
- Do **not** duplicate `skill_packs` into a second catalog table — the registry reads from it (REG-02: no parallel skills-only catalog).

## Step 5 — `tool_search` (pure retrieval + meta-tool)

- Implement `tool_search(query, *, surface=None, capability=None, limit=…) -> list[ToolDefinition]` as **pure retrieval** over the catalog — regex/keyword match on `name` + `description` (+ skill `trigger_tags`). No model call, no usage event. Reuse the scoring shape from `classify()`/`scoreSkill()` so skills and native tools rank consistently.
- Build the **compact catalog** view: `[{name, one-liner}]` for the in-context system-prompt table, capped (≈50), drawn from `always` tools + a compact listing of `deferred` sources.
- Register `tool_search` itself as a `native`, `always` `ToolDefinition` (this is the meta-tool the Phase 3 loop will call). Its executor runs the retrieval and returns matched full definitions (via `to_anthropic`) for the caller to add to the active set.

## Step 6 — Unit-test the core in isolation

- Registration + lookup by name; `to_anthropic`/`to_openai` round-trip a schema identically to the original `KB_EXPLORER_TOOLS`/`SANDBOX_EXECUTION_TOOLS` entries.
- `tool_search` returns expected matches for representative queries; deferred entries not in the compact catalog's full-schema section until searched.
- Skill adapter yields entries for a global skill and an owner's private skill, and excludes another user's private skill.

## Success criteria (this plan)

1. `ToolDefinition` + registry module exist; tools are keyed by name with a `source` discriminator and an executor abstraction covering native-callable and skill-load-by-body (mcp placeholder present).
2. `to_anthropic` reproduces the exact current `KB_EXPLORER_TOOLS`/`SANDBOX_EXECUTION_TOOLS` shapes (verified by comparison); `to_openai` produces valid function-tool shapes.
3. Skills register as deferred entries read from `skill_packs` (no parallel table), with correct global-or-owner visibility.
4. `tool_search` is pure retrieval (no model call), returns matches with full schemas, and is itself a registered native meta-tool; the compact catalog is generated and capped.
5. Unit tests pass; `python -m compileall python-backend` clean.
