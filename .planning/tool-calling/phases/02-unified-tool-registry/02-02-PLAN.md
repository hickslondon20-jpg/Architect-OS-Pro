# Plan 02-02 тАФ Surface/Context Subsetting (D1-Neutral), Citation-Ready Results, Behavior-Preserving Fold-In

**Requirements:** REG-04, REG-05, REG-06
**Depends on:** 02-01 (registry core, sources, adapters, `tool_search`).
**Goal:** Make the registry surface-aware and context-aware via a D1-neutral resolver, give tool results a citation-ready envelope, and prove the whole thing by having the existing sub-agent services source their tools from the registry with identical behavior.

> Read `02-RESEARCH.md` and this phase's `CONTEXT.md` (the "D1-neutral design, concretely" section) first. `agent_capabilities.allowed_tools`/`allowed_surfaces` already exist as the authorization list тАФ the resolver maps them onto registry definitions without fusing the two systems.

---

## Step 1 тАФ The D1-neutral scoping resolver

Add a resolver (e.g. `ToolScopeResolver`) that answers **"which `ToolDefinition`s does surface X / capability Y get?"** with its source-of-truth **injected**:

- Default implementation reads `agent_capabilities` (`allowed_tools` тЖТ registry definitions; `allowed_surfaces` тЖТ surface filter). This is the conservative default and preserves today's authorization exactly тАФ the **two-layers** outcome stays intact.
- The interface must also permit a registry-native surface-tag source (a step toward **one registry**) by swapping the implementation тАФ a one-line change, no caller rewrite.
- **Never** hardcode "registry == capabilities" or "registry ignores capabilities." Document, in code, that this seam is deliberately the D1 fork and must stay swappable (REG-06).

Expose `registry.get_tools(surface=тАж, capability=тАж, names=тАж, format="anthropic")` that runs the resolver then adapts. This never returns one flat global list тАФ a surface/capability scope is required (REG-04). Context-aware subsetting (Ep6 per-phase curated subsets) is satisfied "in shape" by the same `names=`/scope parameters; no Ep6 wiring here.

## Step 2 тАФ The citation-ready result envelope (REG-05)

Define a standard result envelope every tool executor returns, carrying provenance so Ep7 can ground without a retrofit:

- `content` тАФ the tool's payload (as today).
- `sources` тАФ a list of `{source_kind, source_id, verbatim?, label?}` where applicable. Reuse the polymorphic `source_kind`/`source_id` precedent already used by `agent_context_sources`/`artifacts` (no hard FK).
- Tools that are inherently source-grounded populate `sources`:
  - `kb_read` тЖТ `{source_kind:'raw_document', source_id: document_id, verbatim: <read span>, label: name}`.
  - `kb_grep`/`kb_glob`/`wiki_*` тЖТ the doc/page ids + names they already return.
  - `execute_code` тЖТ provenance = the code + stdout (computation is its own evidence), not a document source.
- Tools with no meaningful source carry an empty `sources` list тАФ **never an opaque string** (REG-05).
- This is a *contract + population*, not UI. Ep7 renders it. Keep the envelope minimal and forward-compatible with Ep7's four-tier citation taxonomy (Tier 0 records, Tier 1 wiki, Tier 2/3 docs, web).

## Step 3 тАФ Behavior-preserving fold-in (the proof)

Rewire the existing sub-agent services to **source their tool list from the registry** instead of the hardcoded module constants тАФ without changing behavior:

- `kb_explorer_service.py`: replace `tools=KB_EXPLORER_TOOLS` with `tools=registry.get_tools(capability="kb_explorer_agent", format="anthropic")`; route `_execute_tool` dispatch through the registry executors (or keep the existing dispatch but assert the tool set is identical). Same 8 tools, same schemas, same order, same results.
- `sandbox_execution_service.py`: same, for `execute_code`/`read_skill_file` under `sandbox_execution_agent`.
- The dispatch a tool runs must be identical to today's; wrap each existing executor as the registry `ToolDefinition.executor`. Do **not** add a third parallel dispatch chain тАФ move dispatch to live with definitions and have the services call through it (RESEARCH landmine).
- Wrap each tool's return in the citation envelope (Step 2) тАФ additively; the loop still gets the same `content` it does today, now with `sources` alongside.
- **Do not touch the orchestrator's capabilityтЖТhandler dispatch** beyond what's needed to pass the registry through; that chain is D1-adjacent and stays as-is to keep D1 open.

## Step 4 тАФ Prove D1 both ways (documentation + test)

- Write a short design note (in code docstring + this phase's completion notes) showing how the same registry supports **two layers** (resolver reads `agent_capabilities`, today's default) and **one registry** (resolver reads registry-native tags) by swapping the resolver implementation тАФ nothing else changes. This is the REG-06 acceptance artifact ("documented, not just asserted").
- A unit test that swaps the resolver source and confirms `get_tools(...)` still returns a correctly-scoped set тАФ proving neither outcome is foreclosed.

## Step 5 тАФ Verify no behavior change on the live loops

- Confirm (by comparison / test) that `kb_explorer` and `sandbox_execution` present the exact same tool schemas to Claude after the fold-in as before.
- A live or simulated KB Explorer run returns the same tool steps and result summary as pre-change; the new `sources` envelope is populated for `kb_read` and empty-but-present for `execute_code`.
- Usage tagging (Phase 1) still fires unchanged for these sub-agents.

## Success criteria (this plan)

1. `get_tools(surface/capability)` returns correctly-scoped subsets via an injected resolver; never a flat global list.
2. The resolver's source-of-truth is swappable in one line; a test proves both `agent_capabilities`-backed and registry-native-tag-backed scoping work тАФ D1 not foreclosed (REG-06), documented.
3. Every tool executor returns the citation envelope; `kb_read`/`wiki_*` populate `sources` with source id + verbatim/label; `execute_code` carries computation provenance; no tool returns an opaque string (REG-05).
4. `kb_explorer_service` and `sandbox_execution_service` source their tools from the registry with **identical** schemas, dispatch, and results (behavior-preserving тАФ verified, not assumed).
5. The orchestrator's capabilityтЖТhandler dispatch is unchanged in behavior; D1 stays open.
6. `python -m compileall python-backend` clean; existing sub-agent behavior re-verified live or by simulation.
