# Plan 03-02 — Frontend: Re-point the Chat Client + Live Curated Trace Rendering

**Requirements:** LOOP-01, LOOP-02 (frontend halves).
**Depends on:** 03-01 (the FastAPI SSE endpoint + event vocabulary).
**Goal:** Point the Virtual CSO chat client at the Python streaming endpoint, render the curated thinking/tool trace live as it streams, and cut over behind a feature flag with the Vercel path as rollback — all in the existing design system, no visual regression.

> Read `03-RESEARCH.md`. `lib/virtualCsoApi.ts`'s `parseSseStream` is generic (handles arbitrary `event:` names); `getBaseUrl()` already exists in `skillsApi`/`artifactsApi`. This is a contained re-point + additive rendering, not a rewrite.

---

## Step 1 — Re-point `sendUserMessage` to the Python endpoint (flagged)

- Change the `fetch` target in `sendUserMessage` from the hardcoded relative `/api/vcso/chat` (Vercel) to `${getBaseUrl()}/api/vcso/chat…` (FastAPI), reusing the `getBaseUrl()` pattern from `lib/skillsApi.ts`/`lib/artifactsApi.ts` (`VITE_INGESTION_API_URL`).
- Gate this behind a **feature flag** (e.g. an env/`featureGates` flag `vcso_python_stream`) so the Vercel path remains selectable for instant rollback during transition. Same auth (bearer token from the Supabase session) and same POST body.
- Keep `parseSseStream` — it already supports the new event names.

## Step 2 — Handle the new streamed event types

Extend the `onEvent` handling in `sendUserMessage` (and the `SendUserMessageOptions` callbacks) for the 03-01 vocabulary:

- `ready` — unchanged semantics (user message + `route`/`assembledContext` meta); keep the existing `onReady`.
- `step` / `tool_call` / `tool_result` — **new**: surface a live callback (e.g. `onTrace(step)`) that appends a curated trace entry to the in-flight assistant message as it streams.
- `token` — unchanged (`onToken` appends answer text).
- `done` — unchanged shape (assistant message + sources + sourcePages + artifactId + usage); reuse existing handling incl. `getArtifact`.
- `error` — unchanged.

Reuse the existing `AgentStep` type / `AgentStepsPanel` component (already used for KB Explorer/sandbox steps and for reload reconstruction in `getMessagesForChat`) so live trace and reloaded trace render identically.

## Step 3 — Render the curated trace live in the thread

- Show the streamed `step`/`tool` events **above the streaming answer**, in the existing nested trace panel style (the `AgentStepsPanel` pattern from Ep4 Phase 2), so the founder sees "searching… reading… calculating…" building before/as the answer streams. Curated summaries only — no raw code/payloads (the backend already sends only summaries).
- Keep the visual language in the ArchitectOS design system (Obsidian/Brass/Parchment tokens, no Inter, no default grays) — match the existing trace panel, do not introduce a new component if the existing one fits.
- Ensure **live trace and reloaded trace look the same**: `getMessagesForChat` already rebuilds `agentSteps` from `agent_delegation_runs`/`steps`; confirm the main-loop steps persisted in 03-01 flow through that same reconstruction (this is the Phase 3 ↔ Phase 7 seam — live now, reload-durable formalized in Phase 7).

## Step 4 — Cut over + preserve rollback

- With the flag on, the founder's chat runs entirely against the Python endpoint. With it off, the Vercel path still works.
- Do not delete the Vercel `chat.ts` or its client path in this plan — leave it dormant behind the flag for rollback. Removal is a later cleanup once the Python path is live-smoked.

## Step 5 — Verify parity + streaming feel

- A real turn streams: `ready` → live curated trace entries appear as tools run → answer streams token-by-token → `done` finalizes with sources/artifact.
- Reloading the thread reconstructs the same trace + message (via the existing `getMessagesForChat` path).
- Meta panels (`route`/`assembledContext`) still populate from `ready`.
- No visual regression vs. the current chat.

## Success criteria (this plan)

1. `sendUserMessage` targets the FastAPI SSE endpoint via `getBaseUrl()`, behind a feature flag, with the Vercel path preserved as rollback.
2. The client handles `step`/`tool_call`/`tool_result` in addition to `ready`/`token`/`done`/`error`, appending curated trace live to the in-flight message.
3. Live curated trace renders above the streaming answer in the existing `AgentStepsPanel` style and design tokens — no new component unless the existing one genuinely doesn't fit; no raw code/payloads shown.
4. Live trace and reloaded trace render identically (shared reconstruction path); meta panels still populate.
5. `npm run build` / focused TS check clean; a real turn smoked (or the smoke gap flagged honestly if creds/live backend unavailable in the checkout).
