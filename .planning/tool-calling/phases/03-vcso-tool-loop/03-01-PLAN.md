# Plan 03-01 — Backend: Streaming VCSO Tool Loop + FastAPI SSE Endpoint

**Requirements:** LOOP-01, LOOP-02, LOOP-03, LOOP-04
**Depends on:** Phase 1 (model routing + `usage_events.py`), Phase 2 (`tool_registry` + `tool_search`). Inherits, does not re-touch.
**Goal:** A Python service that runs the Virtual CSO as a genuine Claude tool-use loop consuming the registry, and a FastAPI SSE endpoint that streams curated trace events + answer tokens to the browser — porting `chat.ts`'s orchestration and superseding the Vercel path.

> Read `03-RESEARCH.md` first. The sub-agent loops (`kb_explorer_service`, `sandbox_execution_service`) are the working pattern; extend that shape with the registry, streaming, and delegate/direct paths. No data-model migration is needed for the move.

---

## Step 1 — `VcsoChatService`: the streaming tool loop

Create `python-backend/services/vcso_chat_service.py`, modeled on the bounded native tool-use loop in `sandbox_execution_service.py`, with these differences:

- **Tools come from the registry:** `registry.get_tools(surface="virtual_cso", format="anthropic")` for the always-on core, plus `tool_search` as the discovery meta-tool. **No hardcoded tool list.** The subset is read/compute only (one-writer; no KB-write tool in scope — verify).
- **Context assembly ported from `chat.ts`:** system prompt + WS5 rules + founder wiki compact index/pages + project pinned context + recent messages + prior tool results. Reuse `assemblePrompt`'s section structure; port `loadIpLayer` / `loadFounderContext` / skill index loading to Python (or call existing Python equivalents where they exist — verify before duplicating).
- **Skill routing ported:** explicit `@slug` invocation and `classify()`/`scoreSkill()` ranking → but now a matched skill is a **registry entry loaded via the skill "load-by-body" executor** (Phase 2), not a separate `loadSelectedSkillBodies` path. The pre-flight `shouldCallKbExplorer`/`requires_sandbox` guesses are **absorbed into in-loop `tool_search`** (the model discovers and calls KB/sandbox tools mid-thread). (Retrieval-router-seed decision: see CONTEXT open items.)
- **Model:** resolve `vcso_chat` via `platform_ai_settings` (Claude-locked, Phase 1); guard non-anthropic → Claude fallback.
- **The loop:** each round calls Claude with tools; on `tool_use`, dispatch via the registry executor (direct) **or** delegate to a bounded capability sub-agent through the existing `SubAgentOrchestrator` (delegate) — **both paths supported, neither hardcoded** (LOOP-04, D1-open); feed compact results back; repeat. On `end_turn`, stream the final answer. Cap `max_rounds` with a graceful finish.
- **Sub-agent returns kept compact** (LOOP-05 / Phase 6 protection): synthesized summaries fed into the main thread, not raw dumps.

## Step 2 — Curated trace events (L11) as the loop runs

As the loop executes, emit **curated** trace events (never raw chain-of-thought, never raw tool payloads/code):

- `step` / `tool_call` — a human-readable line ("Searching your knowledge base for retainer structures", "Reading Q3 P&L", "Running the margin calculation"), derived from the tool name + safe input summary (reuse `_safe_input_summary`).
- `tool_result` — a compact outcome summary (reuse `_safe_output_summary`; e.g. "Found 3 documents", "Calculation complete").
- Optionally a short synthesized "plan/why" line — a curated sentence, not the model's raw thinking.

These are the same events that get persisted (Step 4) and that Phase 7 reconstructs.

## Step 3 — FastAPI SSE endpoint

Add the **first** browser-facing streaming endpoint on the Python backend (e.g. `POST /api/vcso/chat` mounted in `main.py`), using `StreamingResponse` (or `sse-starlette` `EventSourceResponse` — pick per CONTEXT open item, matching the frontend's `event:`/`data:` framing):

- Auth: reuse `get_current_user_id` (the existing JWT dependency). Accept the same POST body as today (`threadId`, `text`, `linkedFolder`, `projectId`).
- CORS: ensure the frontend origin is in `config.allowed_origins`.
- **SSE event vocabulary (= Phase 7 persistence contract):** `ready` (threadId + user message + preserved `route`/`assembledContext` meta for UI parity) → interleaved `step`/`tool_call`/`tool_result` → `token` (final answer, streamed via `messages.stream`) → `done` (assistant message + sources + artifactId + usage) → `error`. Always emit a clean `done` on cap/timeout/error.
- Thread/message lifecycle ported from `chat.ts`: create/load thread, insert user message, insert assistant message, update thread counters.

## Step 4 — Persist the trace + relocate `role='main'` usage logging

- **Persist the loop's own tool steps** so Phase 7 can reconstruct them. Lean: reuse `agent_delegation_runs`/`agent_delegation_steps` for the main loop's steps (the client reconstruction path in `getMessagesForChat` already reads these), linking to the assistant message the same way sub-agent runs do. Confirm the shape fits; only add a table if it genuinely doesn't (reuse-before-creating).
- **Emit `role='main'` usage** from the Python loop via `usage_events.py` (it moved out of `chat.ts`). Sub-agent/utility events already fire from Phase 1/2. Tag `surface='virtual_cso'`, `model`, `provider`, `thread_id`.

## Step 5 — Risk posture: parallel endpoint, not a hard swap

- Stand the endpoint up **alongside** the live Vercel `chat.ts` (do not delete it in this plan). The cutover is gated on the frontend (03-02) behind a feature flag.
- Keep the Vercel function dormant/removable as instant rollback.

## Step 6 — `CLAUDE.md` Rule #1 note

Update `CLAUDE.md` Rule #1: VCSO chat streaming now runs in the Python direct-Anthropic lane (streaming to the browser via FastAPI SSE), superseding the Vercel serverless streaming exception for this surface. Keep the N8N and Python-direct lanes otherwise as written.

## Success criteria (this plan)

1. `VcsoChatService` runs a bounded Claude tool-use loop sourcing tools from the registry (no hardcoded list); supports both direct tool calls and sub-agent delegation (D1 not hardcoded).
2. A FastAPI SSE endpoint streams `ready` → curated `step`/`tool_call`/`tool_result` → `token` → `done`/`error`, with the final answer streamed token-by-token; a clean `done` is emitted on cap/timeout/error.
3. Curated trace only — no raw chain-of-thought, no raw tool payloads/code streamed.
4. The tool subset is read/compute only (one-writer); `vcso_chat` resolves Claude-locked; `role='main'` usage emits from Python.
5. The loop's tool steps persist (reused `agent_delegation_runs`/`steps` unless proven insufficient) in a shape Phase 7 can reconstruct.
6. The Vercel path is untouched/dormant (rollback preserved); `CLAUDE.md` Rule #1 updated. `python -m compileall python-backend` clean; unit test of the loop (mocked Anthropic) passes. Live end-to-end smoke flagged if creds unavailable.
