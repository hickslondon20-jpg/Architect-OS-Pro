# Plan 01-01 — Wire Per-Capability / Utility Model Routing

**Requirements:** ROUTE-01, ROUTE-02, ROUTE-03
**Depends on:** nothing (extends live `ai_models` / `platform_ai_settings` / `resolve_platform_model` / `agent_capabilities.model_setting_key`).
**Goal:** Make the *already-existing* capability→model routing seam actually resolve for sub-agents and utility jobs, with the primary Claude chat untouched and current behavior preserved until a row is deliberately changed.

> Read `01-RESEARCH.md` first. The resolver, tables, and `model_setting_key` column already exist and work; the gap is that no `platform_ai_settings` rows exist for capability keys and the sub-agent services hardcode `claude_synthesis_model` instead of resolving.

---

## Step 1 — Seed `platform_ai_settings` rows for active capabilities (migration)

Create a migration (follow the repo's current convention — latest numbered is `012_sandbox_execution_agent.sql`, so `013_capability_model_routing.sql` unless the repo has moved to dated names; verify the latest file first).

For each **active (`experimental`) capability** — `document_analysis_agent`, `structured_data_agent`, `kb_explorer_agent`, `sandbox_execution_agent`, `per_user_wiki`, `per_user_document_wiki`, `global_ip` — insert a `platform_ai_settings` row keyed by the capability's `model_setting_key`:

- `setting_key` = the capability key (e.g. `kb_explorer_agent`)
- `provider` = `anthropic`, `fallback_model_name` = **`claude-sonnet-4-6`** (the confirmed standard), `model_id` → the `anthropic/claude-sonnet-4-6` `ai_models` row, `is_enabled` = true, `settings` = `{}`.
- **Prerequisite:** `ai_models` currently only has a `claude-sonnet-4-5` row (from `005`). This migration must first upsert an `anthropic/claude-sonnet-4-6` `ai_models` row so the capability settings (and `vcso_chat`) can FK to it.
- Use `on conflict (setting_key) do update` so re-runs are safe (mirror `005_metadata_extraction.sql`'s upsert pattern).

**Point them at the confirmed standard (`claude-sonnet-4-6`) on purpose.** This activates the seam and standardizes the whole platform on one Claude version: `chat.ts` is already `4-6`; sub-agents currently default to `4-5` via `config.py`, so this is a deliberate, uniform bump to `4-6` — an intended standardization, not an accidental change. Also bump `config.py`'s `claude_synthesis_model` default to `claude-sonnet-4-6` so the fail-open fallback matches. Cheaper models for specific capabilities are a later config change (a row edit), not part of this phase.

> **Version — RESOLVED (London, 2026-07-02): standardize on `claude-sonnet-4-6`.** Seed the new `anthropic/claude-sonnet-4-6` `ai_models` row, point all capability settings at it, and bump `config.py`'s default to match. This is an interim hardcode of the standard: the forward path is a global **admin settings** area (its own separately-scoped build, deferred) where an admin account picks the current model version from a dropdown that writes to `platform_ai_settings`. Because routing already reads that table, the admin panel later is additive UI, not a schema or wiring change.

## Step 2 — Resolve model via `model_setting_key` in the sub-agent services

In each server-side sub-agent that runs an LLM loop, replace the hardcoded `self.model = settings.claude_synthesis_model` with a resolve call using the capability's `model_setting_key`:

- `services/kb_explorer_service.py` → resolve `kb_explorer_agent`.
- `services/sandbox_execution_service.py` → resolve `sandbox_execution_agent`.
- Any other active sub-agent handler that instantiates an Anthropic model directly (audit `sub_agent_orchestrator.py`'s handlers and `doc_wiki_synthesis.py`).

Use the existing `VectorStore.resolve_platform_model(setting_key=<capability.model_setting_key>, fallback_model_name=settings.claude_synthesis_model, fallback_provider="anthropic")`. Because the resolver fails open, a missing/disabled row keeps today's behavior exactly.

- Prefer resolving from the capability record already loaded in the orchestrator (the capability carries `model_setting_key`) rather than hardcoding the string in the service, so the wiring stays capability-driven.
- If a resolved provider is not `anthropic`, the sub-agent must still function — for beta, keep sub-agent loops on Anthropic and treat a non-anthropic resolution as config error / fall back, since the loop's tool-call/interpret decision is orchestration-grade (ROUTE-02). Document this guard; do not build multi-provider sub-agent execution in this phase.

## Step 3 — Confirm utility jobs resolve through the registry

`MetadataExtractor` already resolves `ingestion_metadata_extraction`. Audit the other utility LLM/embedding calls (embeddings in `vector_store.py`, any title-generation, the guided-skill-draft synthesis `skill_draft_synthesis.py`) and confirm each either already resolves through `platform_ai_settings` or is given a `setting_key` + seed row so its model is registry-driven too. Utility jobs may legitimately run cheaper/specialized models (ROUTE-02) — this is where that becomes real.

## Step 4 — Wire the primary chat to the settings table (Claude-locked) — the forward-compat seam for the admin panel

- **Recommended (elevated from optional):** have `api/vcso/chat.ts` resolve its model from the `vcso_chat` row in `platform_ai_settings` (which becomes `claude-sonnet-4-6` after this migration), with the env/hardcode as fail-open fallback. This is the **forward-compatible seam** for the future global **admin settings** area: when the admin model dropdown lands, it writes the chosen Claude version to `platform_ai_settings.vcso_chat` and `chat.ts` already honors it — no code change then.
- **Still Claude-locked.** Reading a platform-global admin setting is not per-thread/user switching (L12): there is no user-facing model UI, and the setting only ever resolves to a Claude version. Guard so a non-anthropic value falls back to the Claude default rather than switching the primary chat off Claude.
- Document explicitly (code comment + completion report) that when Phase 3 builds the tool loop, the call/interpret decisions run on the Claude chat model — cheap models are only inside sub-agents/utility.

## Success criteria (this plan)

1. Active capabilities have `platform_ai_settings` rows; `resolve_platform_model(capability.model_setting_key, …)` returns a model for each.
2. `kb_explorer_service` and `sandbox_execution_service` (and any other active sub-agent) select their model via the resolver, not a hardcoded field — verified by reading the code, and by confirming a row edit changes the resolved model without a code change.
3. Default runtime behavior is unchanged (same model runs as before the migration) because rows point at the current default and the resolver fails open.
4. No per-thread/user model switching exists for the primary chat; the loop-on-Claude invariant is documented.
5. The platform is standardized on `claude-sonnet-4-6`: an `anthropic/claude-sonnet-4-6` `ai_models` row exists, capability settings + `config.py` default + `chat.ts` all resolve to it, and `chat.ts` reads its model from the `vcso_chat` setting (Claude-locked, no user UI).
