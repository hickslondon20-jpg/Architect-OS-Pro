# Plan 01-02 — The Single Tagged Usage-Event Stream

**Requirements:** METER-01
**Depends on:** nothing structurally (extends live `ai_usage_log`). Sequence after / alongside 01-01 so the same migration pass can carry both if convenient.
**Goal:** One tagged usage-event stream that feeds both the per-thread degradation signal (Phase 6) and the future metering ledger — extending `ai_usage_log` in place, emitting events for every model call (main, sub-agent, utility) across both the TypeScript and Python sides.

> Read `01-RESEARCH.md` first. `ai_usage_log` exists but has no `role` and is written only from `chat.ts` (the `main` call). Sub-agent and utility model calls emit nothing today.

---

## The tag contract (from map §3.4 / L13)

Every model call emits one event tagged: `user`, `thread`, `surface`, `model`, and `role` ∈ {`main`, `sub_agent`, `utility`}.
- **Degradation (Phase 6)** filters to `role='main'` for the current thread.
- **Metering (deferred ledger)** sums *all* roles in a per-user time window.
- A sub-agent's internal work is invisible to degradation, fully visible to metering.

## Step 1 — Extend `ai_usage_log` additively (migration)

In the same migration file as 01-01 (or an adjacent one), add to `public.ai_usage_log`:

- `role text not null default 'main'` with `check (role in ('main','sub_agent','utility'))` — default `'main'` guarantees the existing `chat.ts` insert keeps working unchanged.
- `provider text` (nullable) — e.g. `anthropic` / `openai` / `cohere`, for ledger cost attribution.
- `capability_key text` (nullable) — which capability produced a `sub_agent`/`utility` event (links to `agent_capabilities`; keep as text, no hard FK, mirroring the polymorphic `agent_context_sources` precedent).
- `run_id uuid` (nullable) — optional link to `agent_delegation_runs.id` so a sub-agent's events can be tied to its run. No hard FK required; keep nullable.
- `cost_usd numeric` (nullable) — ledger-readiness placeholder; stays null this phase (no price map built — see build `CONTEXT.md`). Include only if the build-planning checkpoint keeps it; otherwise omit and let the ledger pass add it.

Preserve existing columns and RLS. Confirm `ai_usage_log`'s current RLS/grants and keep the same posture (it's written service-side / via authenticated user context today — verify before changing anything).

## Step 2 — Tag the existing `main` writer

`api/vcso/chat.ts` already inserts into `ai_usage_log` after the assistant message. Add `role: 'main'` and `provider: 'anthropic'` to that insert. No other change to the main path.

## Step 3 — Add a Python usage-logging helper and emit `sub_agent` + `utility` events

Create a small server-side helper (e.g. `python-backend/services/usage_events.py`, or a method on `VectorStore`) that inserts one `ai_usage_log` row with the full tag set. Then call it at every Python-side model call:

- **Sub-agent (`role='sub_agent'`)**: `kb_explorer_service` (per round of its Anthropic loop, or once with summed tokens per run), `sandbox_execution_service`, and any active `sub_agent_orchestrator` handler / `doc_wiki_synthesis`. Tag `capability_key` and, where available, `run_id` + `thread_id`.
- **Utility (`role='utility'`)**: `metadata_extractor` (OpenAI), embeddings in `vector_store`, `skill_draft_synthesis`, any title-generation. Tag `provider`, `model`, `surface` (e.g. `ingestion`), and `user_id`.

Pull `input_tokens`/`output_tokens` from each provider's response usage object (Anthropic `usage.input_tokens`/`output_tokens`; OpenAI `usage.prompt_tokens`/`completion_tokens`). Where a call doesn't return usage, log the event with null token counts rather than skipping it — presence of the event matters for the ledger's call count.

**Emit events best-effort:** a logging failure must never break the underlying call (wrap in try/except, swallow-and-continue), mirroring how `chat.ts` treats its usage insert as non-critical.

## Step 4 — Verify the stream reconstructs both consumers

- **Degradation-ready:** a query filtering `ai_usage_log` to `role='main'` + a `thread_id` returns exactly the main-thread token history (what Phase 6 needs), excluding sub-agent internal usage.
- **Ledger-ready:** a query summing all roles for a `user_id` over a time window returns total token usage across main + sub-agent + utility, grouped by `role`/`model`/`provider` — provable now even though the ledger isn't built.

## Success criteria (this plan)

1. `ai_usage_log` carries `role` (defaulted `main`, checked) + provider (+ optional capability/run/cost fields), added additively with existing rows/RLS intact.
2. The `chat.ts` main insert is tagged `role='main'`.
3. Sub-agent and utility model calls across the Python backend emit tagged events via one shared helper — no parallel table, no second pipeline.
4. Usage-logging failures never break the underlying model call.
5. Both consumer queries above return correct results against real (or simulated) rows: degradation filters to `main`; metering sums all roles by window.
