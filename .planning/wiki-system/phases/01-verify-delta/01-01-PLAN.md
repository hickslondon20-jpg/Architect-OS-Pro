## Verification tasks (each produces a section in 01-01-DELTA.md)

### A. Checkpoint table (the global structured layer)
- Locate the real table(s) behind `lib/gm-audit.ts` — keys observed: `gm_checkpoint_id`,
  `checkpoint_id_display` (e.g. "1.1.1"), `checkpoint_title_display`, capability/screen mapping.
- Run against Supabase:
  `SELECT table_name FROM information_schema.tables WHERE table_schema='public' AND (table_name ILIKE '%checkpoint%' OR table_name ILIKE '%gm_%' OR table_name ILIKE '%mra%');`
- For each candidate: dump columns + row count. **Determine whether the 5-stage AE-Ladder
  calibration (125 capability checkpoints × 5 stages = ~500 rows) is physically stored**, or
  whether the stage dimension is applied at query time / joined from another table.
- Record the exact query interface the `structured_data` tool and the stage-primer will use
  to fetch "what good looks like at stage N for capability X."
- **Delta call:** confirm or correct CONTEXT §5.1 and the spec's `mra_checkpoints` guess.

### B. Sub-agent orchestrator hosting
- Read `sub_agent_orchestrator.py`, `agent_capabilities.py`, `agent_context.py`,
  `structured_data.py`, `structured_query.py`.
- Confirm new capabilities (`per_user_wiki`, `global_ip`) can register as rows in
  `agent_capabilities` with `allowed_tools`, `allowed_surfaces`, versioned `output_schema`,
  and dispatch by handler in `start_run` — matching the existing
  `document_analysis_agent` / `structured_data_agent` / `kb_explorer_agent` pattern.
- Document the dispatch handler insertion points and any rework (e.g. new `allowed_source_kinds`
  like `wiki_claim`, `wiki_page`, `global_ip_page`).
- **Delta call:** confirm CONTEXT L10 ("no new plumbing") or list the plumbing required.

### C. Existing wiki UI scaffolding
- Read `components/pro-suite/os-engine/views/WikiView.tsx` and its siblings
  (`IndexView.tsx`, `StructureRail.tsx`, `kb/FileNode.tsx`, `OSEngineWorkspace.tsx`).
- Document its current data model and how the three-class page model (compiled / insight /
  override) and claim/evidence rendering map onto it. Note what is mock (`lib/osEngineMockData.ts`)
  vs. wired.
- **Delta call:** what the UI already gives us vs. what the schema must expose to it. (UI
  rebuild is NOT in this build's scope — this is mapping only.)

### D. Provenance UI
- Read `components/pro-suite/virtual-cso/SourcesPanel.tsx`. Confirm it can render the
  `evidence[]` shape (`source_id, source_kind, path, lines, weight, note`). Note gaps.

### E. Pre-existing wiki tables
- `SELECT table_name FROM information_schema.tables WHERE table_schema='public' AND table_name ILIKE '%wiki%';`
- Inventory any existing `wiki_*` tables / columns / RLS. Flag anything reusable or conflicting
  before sub-phase 03 writes migrations.

### F. Tier 0 source inventory (for the event→rebuild map)
- Identify the Supabase tables that feed each compiled-base page (diagnostic results, AE Ladder
  scores, sprint/quarter state, Clarity Compass, financial/client uploads). List table names so
  sub-phase 04 (compilation) can wire the event→rebuild map to real change sources.
