Implement the narrow structured mutation surface (B6) and the domain-agent write-back path.

touching the compiled base, with founder-confirmation as the only promotion path.

## Build

### Mutation endpoints (each writes `wiki_action_log`)
```
propose_insight_claim(user_id, page_key, text, evidence[], confidence)
  - page must be insight_accreting (L8); reject on compiled_base_only pages
  - run D11 gates: about_business_ok, novelty_ok (dedup vs existing claims/insights),
    confidence_bar_ok (>= medium, L9)
  - on pass: insert class='insight', status='quarantined' claim + wiki_insight_records row
    (origin='domain_agent_writeback', trust_state='quarantined')
  - on fail: log rejection reason; write nothing to claims
set_claim_confidence(user_id, claim_id, confidence)     -- not on compiled base unless via compile
flag_contradiction(user_id, claim_id, against_claim_id|page_ref, note)
  - inserts wiki_contradictions; sets contested where applicable; records BOTH positions (A3)
add_override(user_id, page_key, claim_id?, text)        -- FOUNDER ONLY; class='override';
                                                           may set superseded_by on a compiled claim
promote_insight(user_id, insight_id)                    -- FOUNDER-CONFIRMATION ONLY (D10/2a):
                                                           trust_state -> trusted, claim.status -> trusted;
                                                           NOT agent-callable; reversible (B4) via demote
demote_insight(user_id, insight_id)                     -- reverses a promotion using action-log payload
```

### Actor-scope enforcement (hard)
- Domain agents may call only `propose_insight_claim` (+ `flag_contradiction`).
- Founder may call `add_override`, `promote_insight`, `demote_insight`.
- No actor may write `class='compiled'` (enforced by the 03-01 trigger + here at the API).

### Write-back trigger = session-end / pre-compaction flush (B5)
A silent pass at session end proposes write-backs so insights aren't lost on context
compaction. Wire this into the Virtual CSO session lifecycle **as a producer of proposals
only** — it calls `propose_insight_claim`; it never promotes. (The actual chat integration
that *invokes* this lands in the connection phase; here, build the flush entry point + the
proposal API it calls, testable in isolation.)

### Reasoning-only enforcement (D9)
Quarantined insight claims are returned by reads with `trust: quarantined` so the orchestrator
may let them shape its line of inquiry, but the contract + read handler must make them
**non-assertable** — document the convention and add the flag the synthesis layer (connection
phase) will honor. Two-gate hallucination defense: (a) cannot be asserted, (b) cannot reach
trusted without founder sign-off.

---

## Success criteria
1. All five mutations enforce actor-scope; unauthorized actor/class is rejected.
2. `propose_insight_claim` runs all three D11 gates and only quarantines on pass.
3. Promotion is founder-only, the sole quarantined→trusted path, and reversible via the action-log.
4. Every mutation appends an action-log row with before/after payload.
5. Session-end flush entry point exists and calls `propose_insight_claim` (no promotion).
6. Compiled base is unreachable by every endpoint here.

## Out of scope
Compilation (sub-phase 04), dreaming (sub-phase 07), health surfaces (sub-phase 06), live chat invocation (connection phase).
