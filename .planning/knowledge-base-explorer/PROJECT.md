# Knowledge Base Explorer — ArchitectOS Pro

## What We're Building

A Claude Code-inspired exploration layer that gives ArchitectOS AI agents (Virtual CSO, Domain Agents, OS Engine) filesystem-like tools to navigate, search, and read a hierarchical knowledge base stored in Supabase. Founders organize uploaded documents into nested folders; the agent can explore this structure using ls, tree, grep, glob, and read tools — the same way Claude Code explores a codebase.

## Core Value

The agent can navigate founder-uploaded documents the same way Claude Code navigates source files — not just flat retrieval, but structured exploration of organized context.

## Two-Layer Data Model

This build addresses **Layer 1 only: user-uploaded documents.**

| Layer | What | How agent accesses it |
|---|---|---|
| **Layer 1 (this build)** | User-uploaded files (PDFs, DOCX, Excel, transcripts, SOPs, research) + platform-synthesized wiki/markdown docs | KB Explorer tools (ls, tree, grep, glob, read) |
| **Layer 2 (existing)** | Structured founder context (MRA results, AE Ladder, sprint plans, brand briefs, initiatives) | Direct Supabase queries from agent context assembly |

Layer 2 data stays in its own Postgres tables. It is NOT part of the KB folder tree. Agents pull from those tables directly. The KB Explorer does not replace or wrap structured data access.

## What Lives in the KB Folder Tree

- Documents founders upload: PDFs, DOCX, XLSX, transcripts, SOPs, market research, client materials
- Platform-synthesized wiki/markdown documents generated by the backend from structured data
- Chat synthesis artifacts (previous conversations synthesized and stored as surfaceable documents)

## What Does NOT Live in the KB Folder Tree

- MRA results, AE Ladder profiles, sprint plans, initiatives, brand briefs (Supabase tables — direct query)
- IP lenses and skill packs (global agent-only content — separate architecture, separate iteration)
- Raw chunked embeddings (internal — not user-facing)

## Folder Visibility Model

**Per-user only.** Founders create and manage their own folder hierarchy. There are no user-visible global folders. Global agent-only content (IP lenses, platform IP) exists in separate Supabase storage and wiki architecture — it is out of scope for this build and will be addressed in its own iteration.

## Stack Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Backend for folder CRUD + KB tools | Python/FastAPI (`python-backend/`) | Existing service, consistent with phases 4–7 when tools need folder logic |
| Deployment | Railway (existing) | N8N already there; Python backend deploys to same environment |
| Data layer | Supabase Postgres + pgvector | Already in use; RLS handles per-user isolation |
| Storage | Supabase Storage (`raw-documents` bucket) | Already set up in intelligence layer architecture |
| Embeddings | OpenAI `text-embedding-3-small` | Ingestion infrastructure exception; not synthesis |
| Metadata extraction | OpenAI `gpt-4o-mini` | Ingestion infrastructure; already in use |
| Observability | LangSmith | Already wired in Python backend |
| Folder visibility | Per-user only | No team accounts in beta; global IP is separate architecture |
| Naming convention | `kb_` prefix | Already in use in codebase; clarity over creativity |

## Constraints

- **No global user folders.** The schema must not include a global/shared tier for user-created content.
- **Preserve existing UploadsView.tsx.** Phase 3 (Ingestion UI) must co-create with existing layout — not overwrite it.
- **Per-phase alignment.** Each phase requires a discuss+approve checkpoint before execution begins.
- **No team accounts in beta.** All folder logic is single-founder scoped.
- **Context window discipline.** Tree/ls output must use depth limits and truncation to protect agent context.
